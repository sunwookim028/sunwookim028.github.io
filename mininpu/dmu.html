DMU: Data Moving Unit - Design Brief

Consolidated, doc-first design brief for the compute-tile async DMA unit. It assembles already-locked decisions and the existing DMA RTL surface into one authoritative contract; it does not invent new microarchitecture. Everything is split into LOCKED (the contract the DMU must honor) and OPEN (the implementer's design space).

00 How to read this brief

Two buckets, marked on every claim. Design inside OPEN; do not re-open LOCKED.

Locked

Contract the DMU must honor. Fixed by the charter (SSOT), the ISA SSOT, and the existing RTL surface already on main. Changing any of these is out of scope for the implementing session; it would ripple into the ISA, the ABI, or the compiler contract.

Open

The implementer designs this. Per session-decision §M the DMU internals (burst sizing, FIFO / landing-buffer depth, dm_axi error/timeout policy, slot FSM, the precise SPAD arbitration handshake, and the exact descriptor encoding) are deliberately deferred to RTL implementation. Every OPEN item below lists the constraints it must satisfy, so the choice is bounded, not blank.

Source anchors are cited as file:line against the tree at origin/main. Primary SSOT: charter §4-§5 (docs/charter.md), ISA (mininpu/isa/opcodes.py, mininpu/isa/encoders.py), hardware parameters (mininpu/hwparams.py), and the DMA RTL under npu/src/.

01 Role & Doctrine

Where the DMU sits and the async model it implements.

Locked · placement

The DMU lives in the compute tile, following the Gemmini LoadController model. The interface tile stays devmem-only: it owns the AXI path to off-chip memory, not the SPAD-fill orchestration. Charter §5: "The DMA unit lives in the compute tile (Gemmini LoadController model); the interface tile stays devmem-only." The sequencer keeps the DMU external to itself: tile_sequencer exposes dma_slot_done / dmu_idle stubs and dmu.sv is a separate module (session-decision §J).

Locked · async dual-port ping-pong

The DMU is async-only (session-decision §A; ISA Doc 1 §4). An async DMA request runs in the background, filling SPAD while the sequencer computes on the compute-side SPAD port (dual-port ping-pong). Charter §5: "an async DMA request (descriptor flag) runs in the background, filling SPAD port A while the sequencer computes on port B." The compute pipeline never stalls for the DMU except at an explicit flush.slot fence: the DMU absorbs all SPAD arbitration variability by yielding (session-decision §I).

Locked · completion token

Completion is signalled by dma_slot_done[slot], the VTA-style dependence token. flush.slot is the fence that waits on it. Charter §5 / §4: "Completion = dma_slot_done[slot] (the VTA-style dependence token); flush.slot fences" and "async DMA + dma_slot_done IS in v1 ... must be RTL-verified."

Sync vs async

  • Async is the only DMU mode in v1. The former async-mode flag (flags[5] on the DMA opcodes) is retired and is now reserved-must-be-zero (ISA Doc 1 §4; session-decision §A). There is no synchronous DMU opcode variant.
  • The host-facing synchronous block transfers (host<->devmem memcpy) are a separate concern, handled by the command processor + copy engine over the AXI-Lite CMD window (charter §4), not by the compute-tile DMU.

02 ISA Surface

The two DMU opcodes, their operand slots, the fence, and the slot count. All LOCKED (ISA SSOT).

Locked · no ACC

The DMU never touches ACC. The dmu.*.acc opcodes are removed (no OP_DMU_*_ACC in the SSOT). ACC is MXU-owned fp32 and never reaches DRAM; async DMA targets the SPAD region only and flush.slot guards SPAD. opcodes.py:232: "The DMU touches SPAD only (ACC never reaches DRAM)."

64-bit encoding, operand slots

The instruction word is the fixed 64-bit format OP[63:56] | FLAGS[55:48] | op0[47:32] | op1[31:16] | op2[15:0] (ISA Doc 1 §3). The DMU opcodes use these slots (encoders.py:436-548):

fielddmu.load.spm (0x40)dmu.store.spm (0x42)
op0SPAD dst rowSPAD src row
op1literal-pool index of dram_addr (IMM_OP1)literal-pool index of dram_addr (IMM_OP1)
op2(mat_rows-1)<<8 | (mat_cols-1), packed 7+7packed shape, or 0xFFFF "derive at lowering" sentinel
pool adjacencydram_stride is force-appended at pool[op1+1] so the descriptor-lowering pass (W6) reads addr + stride contiguously
flagsIMM_OP1 set (address is wide); dtype in base flags; flags[5] reserved-must-be-zero (retired async flag)
  • dram_stride = bytes-per-DRAM-row. Row r reads dram_addr + r*dram_stride. dram_stride=0 is a first-class broadcast mode on load (all dst rows filled from one DRAM row - this is how bias is added). encoders.py:467-490.
  • Inline form only; the W6 assembler.py pass lowers the inline op into a descriptor + trigger. The exact descriptor format the DMU consumes is OPEN (see §07).
Locked · slots & fence

DMA_SLOT_COUNT = 2 independent async slots (hwparams.py:80). flush.slot takes a 16-bit slot bitmask in op0: bit i set = wait for slot i to drain; flush.slot 0xFFFF waits for every slot. It is the only fence primitive (the unmasked flush 0x52 is retired). halt drains all outstanding slots then asserts done. encoders.py:75-88.

03 Data Path & Memory

devmem, the DM word, the AXI bridge, the bandwidth story, and SPAD arbitration priority. All LOCKED.

device memoryoff-chip PL-DDR4, reached via dm_axi_bridge → m00_axicharter §5

Locked · bandwidth - bus width is the lever

The as-built dm_axi_bridge is 32-bit AXI @ 50 MHz. The bottleneck is bus width, not outstanding depth. At 32b/50 MHz the bandwidth-delay product is ~1 burst, so multiple-outstanding buys ~nothing until the AXI path is widened (toward the 256-bit DM-word width) and/or clocked up. Charter §6 notes the DDR/AXI clock is independent of the compute-tile clock (a CDC at the DMEM boundary lets the AXI side run at the MIG user clock, 200-300 MHz), which is the lever that makes the widened path real. This bounds v1 to single-outstanding (see §06).

Locked · SPAD arbitration priority

SPAD has a compute-side port and a DMU access path. On any structural conflict, compute has fixed priority and the DMU yields. Because the DMU is async and flush.slot-gated, yielding never stalls compute. Session-decision §I: "DMU ABSORBS arbitration variability (yields on conflict; it is async/flush-gated), so compute never stalls. Reuse the banked_sram arbiter, priority compute(fixed) > DMU(yields)." The precise handshake (dedicated port A fill vs. a lower-priority compute master on the banked_sram grant arbiter) is OPEN - see §04 and §07.

04 Existing-RTL Surface (reuse, do not rebuild)

Four modules already on main provide most of the mechanism. The DMU orchestrates them; it does not re-implement them.

Note · RTL-vs-target framing

The as-built dma_engine.sv lives in the interface tile and decodes the host block-transfer modes (stream write/read, IRAM load); its 2-slot machinery and dma_slot_done are the completion mechanism the DMU reuses. The target DMU is a compute-tile module (dmu.sv) that drives dma_addr_gen + the dm_* bridge port and the SPAD DMA path. Reuse these ports and the 2-slot/grant primitives; do not rebuild them. How the slot machinery relocates or is re-instantiated inside the compute tile is an implementation choice (§07).

05 Interfaces the DMU Module Must Expose

Shape is LOCKED; exact bit-encoding is OPEN. The DMU sits between the sequencer, SPAD, and the dm_* bridge port.

1. Sequencer descriptor-in

fieldmeaningstatus
baseDRAM base byte address (row 0), from the literal poolLOCKED present
SPAD rowdst row (load) / src row (store)LOCKED present
length (rows × cols)mat_rows, mat_cols (packed 7+7 in op2)LOCKED present
stridebytes-per-DRAM-row; 0 = broadcast (load)LOCKED present
directionload (0x40) vs store (0x42)LOCKED present
slotwhich of the 2 async slots this transfer occupiesLOCKED concept, OPEN encoding

The field set is fixed by the ISA (§02). The exact wire format of the descriptor the sequencer hands the DMU (bit packing, how slot is assigned/carried, how the pool-indexed base/stride are resolved to values) is OPEN (§07).

2. SPAD port + arbitration handshake

  • LOCKED shape: a SPAD read/write access path (256-bit rows, MEM_LATENCY=2) with a yield-on-conflict handshake; compute has fixed priority.
  • OPEN detail: whether the DMU uses the dedicated Port A fill path or is wired as a lower-priority master on the Port B banked_sram grant arbiter, and the exact en/gnt/retry protocol (§07).

3. Completion out + clear

  • LOCKED: drive dma_slot_done[1:0] (sticky, one bit per slot) on transfer completion; accept a clear_slot_done[1:0] write-1-to-clear. This is exactly the dma_engine.sv contract (dma_engine.sv:87-89) and it must be RTL-verified in v1 (charter §4).

4. flush.slot handshake

  • LOCKED: expose per-slot status so the sequencer can masked-stall on flush.slot (op0 bitmask) and drain all slots on halt. The sequencer already stubs dma_slot_done / dmu_idle for this.
  • OPEN: the exact busy/idle signalling and the stall-release timing contract (§07).

5. dm_* to the bridge

  • LOCKED: drive the dm_addr / dm_din / dm_dout / dm_en / dm_we / dm_ack BRAM-style port of dm_axi_bridge (dm_axi_bridge.sv:35-40); one DM word per handshake, dm_ack ends the transaction. The AGU byte address feeds dm_addr via its [DM_BYTE_SHIFT +: DM_ADDR_WIDTH] slice.

06 v1 Scope vs Deferred

What v1 must deliver, and what is explicitly out of v1.

Locked · v1 = make-async-real, single-outstanding

v1 makes the async path real and RTL-verified: dma.load.spm / dma.store.spm run in the background, complete via dma_slot_done[slot], and are fenced by flush.slot. v1 is single-outstanding: one transfer in flight hides the latency. The two slots (DMA_SLOT_COUNT=2) exist in the HW contract, but v1 does not exploit multiple-outstanding, because at 32b/50 MHz the BDP is ~1 burst (§03). Charter §5: "[v1] = make-async-real with single-outstanding (one slot in flight hides latency)."

Deferred · [v1+], out of v1

Multiple-outstanding + coalescing + a landing/reorder buffer are [v1+], layered only after the AXI path is widened toward the 256-bit DM word. The landing/reorder buffer is sized to the bandwidth-delay product (= target BW × DDR latency). Charter §5 / §13. These are explicitly not in v1 and the v1 DMU must not be blocked on them - but its slot/descriptor structure should not preclude them.

07 OPEN Design Space

The implementer's job (session-decision §M). Each item is bounded by the constraints it must satisfy - design within them.

OPEN itemconstraints it must satisfy
Descriptor format / encoding Must carry the §05 field set (base, SPAD row, rows×cols, stride, direction, slot). Must be producible by the W6 assembler.py descriptor-lowering pass from the inline 0x40/0x42 form, reading dram_addr at pool[op1] and dram_stride at pool[op1+1]. Broadcast (stride=0) must be representable.
Slot FSM 2 slots (DMA_SLOT_COUNT=2), each IDLE→ACTIVE→DONE. Must drive sticky dma_slot_done[i] until clear_slot_done[i]. Must support the flush.slot masked fence and full drain on halt. v1 single-outstanding, but the structure should not preclude [v1+] multiple-outstanding. May reuse the dma_engine.sv round-robin fairness.
Burst sizing Bounded by the bridge: one DM word = one 8-beat INCR burst, 256-bit word, byte addr = dm_addr×32. Row loop iterates base + row*stride via dma_addr_gen. Must respect the single-outstanding, in-order, single-ID bridge (no reordering, one transaction to dm_ack).
FIFO / landing-buffer depth v1 single-outstanding needs only enough buffering to bridge SPAD-side (MEM_LATENCY=2) and the AXI burst. A landing/reorder buffer sized to the BDP is [v1+] (after AXI widening). Do not oversize for v1; do not preclude the [v1+] buffer.
dm_axi error / timeout policy The bridge accepts/ignores rresp/bresp today (single-ID, in-order). Define how the DMU handles a stalled/never-acked transaction and any error response. Must not silently corrupt SPAD or hang the flush.slot fence. Sim target is a clean cocotbext-axi AxiRam (charter §5), so the policy is about robustness, not a modeled fault.
Precise SPAD arbitration handshake Compute fixed priority > DMU yields; compute never stalls; the DMU absorbs the variability (async / flush-gated). Reuse the banked_sram grant arbiter (comp_gnt / comp_conflict_bank_o) or the dedicated Port A path. Define the en/gnt/retry protocol and how a yielded DMU beat is re-driven. 256-bit rows, MEM_LATENCY=2.

08 Block Diagram Sketch

Structural layout (information + placement fixed here; aesthetic polish is a later session's job). Descriptor slots → AGU → burst/AXI engine → PL-DDR4, plus SPAD-port arbitration and the dma_slot_done / flush.slot token path.

  • Green = existing/external RTL reused as-is (interface tile bridge, PL-DDR4, the SPAD TDP macro, the compute masters). Teal = the DMU the session builds; boxes marked (OPEN) are the design space.
  • Dashed amber = the dependence-token path: dma_slot_done[slot] back to the sequencer where flush.slot consumes it; clear_slot_done is write-1-to-clear.
  • The SPAD arrow is deliberately drawn as a single "fill / drain" path: whether it is the dedicated Port A or a yielding Port B master is OPEN (§07).

09 For the Implementing Session

A short checklist: what to read, what is fixed, what to decide, what to test.

Read first

  • docs/charter.md §4-§5 (the SSOT doctrine).
  • mininpu/isa/opcodes.py + mininpu/isa/encoders.py (0x40 / 0x42 / 0x53 / 0x5F encodings).
  • The four RTL modules in §04: npu/src/interface_tile/dma_engine.sv, npu/src/interface_tile/dm_axi_bridge.sv, npu/src/compute_tile/dma_addr_gen.sv, npu/src/compute_tile/spad.sv (and npu/src/common/banked_sram.sv for the grant arbiter).
  • The tile_sequencer dma_slot_done / dmu_idle stubs (the DMU is kept external).

Fixed (do not re-open)

  • Placement (compute tile), async-only, SPAD-only (no ACC).
  • Opcodes 0x40 / 0x42, flush.slot fence, dma_slot_done[slot] completion, DMA_SLOT_COUNT=2.
  • devmem = PL-DDR4 via dm_axi_bridge→m00_axi; 256-bit DM word; compute-fixed > DMU-yields SPAD priority.
  • v1 = make-async-real, single-outstanding; multiple-outstanding/coalescing/landing-buffer are [v1+].

Decide (the OPEN list, §07)

  • Exact descriptor format/encoding; slot FSM; burst sizing; FIFO / landing-buffer depth; dm_axi error/timeout policy; the precise SPAD arbitration handshake.

Tests to write

  • dma.load.spm round-trip: preload a cocotbext-axi AxiRam on m00_axi, run dmu.load.spm, assert SPAD rows match (including a stride=0 broadcast case and a multi-row strided case).
  • dma.store.spm round-trip: write SPAD, run dmu.store.spm, assert AxiRam contents match (bf16, row-major, strided).
  • dma_slot_done token: assert the sticky per-slot done bit asserts on completion and clears on clear_slot_done (write-1-to-clear); RTL-verified per charter §4.
  • flush.slot fence: a compute op reading a DMU-filled SPAD row after flush.slot <mask> must observe the completed data; before the fence it must not be relied upon. Verify masked (single-slot) and full (halt) drain.
  • Non-stall: with a DMU transfer in flight, compute Port B access proceeds every cycle (DMU yields on bank conflict).
Reminder

Per the repo DV gate, run the relevant simulation after any RTL change and (if EXEC_DISPATCH is touched) python3 dv/scripts/check_isa_coverage.py. The DMU opcodes already exist in the ISA SSOT, so this brief implies no ISA change - only new RTL + DV.
