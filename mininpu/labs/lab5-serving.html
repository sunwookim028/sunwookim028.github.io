    Lab 5: Key-Value Cache Management¶

    ECE 4950/6950 AI Hardware, Fall 2026

    School of Electrical and Computer Engineering, Cornell University

    Lab 5: Key-Value Cache Management

    Unverified claims on this page.

    • Backend API and semantics: backend.kv_append(kv_k, kv_v, k_new, v_new, seq_pos) writing the new key and value rows at seq_pos, and backend.decode_attn(q, kv_k, kv_v, n_ctx, ...) attending over exactly the first n_ctx rows.

    • Block granularity: the sixteen-rows-per-block figure in Part 2 is written as an example rather than a requirement; confirm no fixed hardware block size contradicts it.

    Introduction¶

    In Lab 4 you wrote the kernels that run on the accelerator. In this lab you build the software around them that serves a language model: the layer that runs the model token by token and manages the memory it needs to do so. The subject of the lab is the key-value cache, the store of past keys and values that lets a model generate each new token without recomputing the whole prompt every time.

    A decoder language model generates text one token at a time. To produce the next token it runs the transformer over the sequence so far and reads off a prediction. The simple way to do this re-runs the model over the entire sequence at every step, which repeats almost all of the work, because the keys and values for the earlier tokens are the same every time. The fix is to keep those keys and values in a cache and extend it by one row per step, so that generating a token costs work proportional to the current length rather than to its square. Building and managing that cache is what this lab is about, first as a single contiguous store, then as a paged store that does not need one contiguous block per sequence.

    This lab is software only, and it runs on the host backend. You write Python that calls the kernels; the serving logic you build does not depend on which backend executes underneath it.

    There are no new instructions to add. The hardware you finished in Lab 3 already reads and appends a key-value cache, and the instruction set is fixed, so everything in this lab is built from the kernels and the operations that already exist. The work is entirely in the serving logic on top of them.

    The course introduction document (labs_introduction.pdf) describes the accelerator, its programming model, the setup on the ecelinux servers, and how submission and grading work. Read it first; this handout assumes it.

    Prerequisites. Lab 4 (the kernels you call here) and Python. You should understand autoregressive decoding at a high level: given a prompt, the model generates one token at a time, appending each generated token to the sequence before generating the next. No prior experience with serving systems is assumed.

    By the end of the lab you will understand why the key-value cache exists and what it saves, how a paged cache removes the requirement that a sequence occupy one contiguous block of memory, how a block allocator and a simple scheduler hand blocks to sequences and reclaim them, and what it takes to offload cold parts of the cache to host memory when on-device memory runs short.

    Materials¶

    Your lab folder contains a working reference engine and a suite of tests. As in the earlier labs, each part is a region you fill in, marked with a # TODO(lab5) comment,

    inside otherwise complete code. The pieces are:

    • The engine you complete, under engine/. The decode loop that drives generation, and the logic that picks the next token, are provided and work; you fill in the key-value management inside them.

    The cached decode loop is greedy_decode in engine/engine.py, and the cache itself is managed in engine/kv_store.py. Each TODO region states what it must compute. - The provided kernels, the key-value append and the cached attention, under kernels/. The append kernel, kv_cache_append in kernels/kv_cache.py, writes the new key and value rows into the store. The attention kernel, decode_attn in kernels/decode_attn.py, computes attention for the single new token over the cached keys and values. Both are complete and tested, and you call them through the backend without modifying them. They are the same kind of kernel as in Lab 4. - The reference engine, the recomputing decoder. This is generate in engine/engine.py: a simple, obviously correct implementation that re-runs the full forward pass over the growing prefix on every step. It is slow and wasteful, but it is the definition of correct output. Your cached engine must produce exactly the same tokens it does. - The test models, under models/, a small configuration for fast iteration and a full model for validation. The worked model is GPT-2. Decoding is deterministic (greedy), so a correct engine reproduces the reference token for token. - The tests, under tests/, written with pytest. make test runs all four parts; make help lists the targets.

    Everything else in the repository is provided and complete; you edit only the # TODO(lab5) regions named above.

    After cloning and setting up the environment as described in the introduction document, install the package and run the tests once:

    pip install -e .
make test

    The tests fail until the TODO regions are filled. The autograder runs the same tests under the same pinned versions, so the result you see locally is the result you are graded on.

    Goal¶

    You build the serving engine in four parts. Parts 1 and 2 build a cached decoder and then a paged cache for it. Parts 3 and 4 add a block allocator with a simple scheduler and a path that offloads cold cache blocks to host memory. Each part builds on the one before it.

    Part 1: cached decode¶

    The first part builds the cached decoder. Instead of re-running the model over the whole sequence at each step, you keep a key-value cache and extend it. The engine tracks how much of the cache is filled in a position counter, pos, which starts at the prompt length and advances by one per decoded token. You maintain the key and value caches as contiguous tensors, pre-allocated to the maximum sequence length, and on each decode step you call two provided kernels through the backend. First the append kernel, backend.kv_append(kv_k, kv_v, k_new, v_new, seq_pos): it writes the current token's key row k_new and value row v_new into the caches kv_k and kv_v at seq_pos = pos. Then the attention kernel, backend.decode_attn(q, kv_k, kv_v, n_ctx, ...): it takes the single new token's query q, the two caches, and the filled length n_ctx, and computes attention over exactly the first n_ctx rows. You pass pos as the append position and the filled length as the attention's n_ctx.

    The reason this works is that the keys and values for past tokens do not change, so they are computed once and reused. At step t the query is the one new token's embedding and the keys and values are the t rows already in the cache; the cost of a step grows with the current length, not with its square. This is the central optimization in any serving system. Because decode_attn attends over exactly the first n_ctx filled rows, it never reads cache slots that have not been written yet.

    Part 1 is graded by test_cached_decode.py,

    which runs greedy decoding on the small model and requires the generated tokens to be exactly those the reference engine produces. The grading is bit-exact: every token must match, because the reference and your engine compute the same arithmetic and differ only in whether work is reused.

    Part 2: a paged cache¶

    Part 2 replaces the contiguous cache with a paged one. A contiguous cache must reserve a single block of memory large enough for the longest sequence it might ever hold, which wastes memory when sequences are short and varied. A paged cache instead stores the keys and values in fixed-size blocks (for example sixteen rows per block) drawn from a shared pool, and keeps a block table: a list, per sequence, of which physical blocks hold its logical positions. When a sequence grows past the end of its current block, you take a free block from the pool and append it to the table; when the sequence finishes, you return its blocks to the pool.

    The point is that paging decouples a sequence's length from a contiguous allocation, in the same way virtual memory decouples a process's address space from contiguous physical memory. Logical position p lives in block ⌊p/B⌋ of the table at offset p mod B, where B is the block size; the blocks themselves need not be adjacent. The attention call takes the block table in place of a contiguous pointer and gathers the keys and values through it. This is the design used by recent serving systems such as vLLM and SGLang, which call it paged attention.

    Part 2 is graded by test_paged_kv.py,

    which requires the same bit-exact greedy output as Part 1 and additionally checks that the pool's free blocks are all reclaimed after a sequence completes.

    Part 3: allocator and scheduler¶

    Part 3 factors the block management into a standalone allocator and adds a simple scheduler over it. You implement the allocator, which owns the pool of free blocks behind a free list and exposes three methods: allocate(n) returns a list of n free block indices and removes them from the pool; free(blocks) takes a list of block indices and returns them to the pool; and available() returns the number of free blocks remaining as an integer. The scheduler holds incoming requests in a queue and admits the next one by calling available() and then allocate(n) once the pool holds enough blocks for it; a queue item is a pending request carrying the number of blocks it needs. One request runs at a time, and the scheduler serves them first come, first served. This allocator interface is the same one Part 4 builds on, so define it cleanly here.

    The point of this part is the interface between the memory manager and the scheduler, not throughput. A production scheduler that runs many requests at once and reclaims blocks continuously is considerably more involved; here the goal is to see clearly how a scheduler asks the allocator for memory and waits when there is not enough. It is graded by test_allocator.py, which exercises allocation and freeing including deliberate over-allocation, and test_scheduler.py, which runs a queue of requests through a small pool and checks that they are ordered correctly and that blocks are reclaimed between them.

    Part 4: offload to host memory¶

    Part 4 adds an offload path for when the on-device pool fills up. When no free blocks remain, you move the coldest cache blocks out to host (PS-DDR) memory to make room, and fetch them back before they are needed for attention again. The allocator interface from Part 3 is given, and two host-memory calls name the transfer endpoints: offload(blocks) copies the listed on-device blocks out to host memory and frees their on-device slots, and fetch(blocks) copies the listed blocks back from host memory into freshly allocated on-device slots. What those two calls move is fixed; the choice of which blocks to evict and the transfer logic that drives them are yours to design. A reasonable eviction policy is least-recently-used; if you choose another, justify it in the report. The idea of trading capacity against a fetch cost is the one FlexGen uses to run models far larger than GPU memory on a single GPU, by spilling weights and cache to host memory and disk.

    The point is that offloading extends the usable cache beyond on-device memory, at the cost of a fetch when an offloaded block is needed again. How much to offload, and when to fetch a block back before it is needed, is the trade-off, and it is the subject of the report for this part. It is graded by test_offload.py,

    which runs a sequence long enough to exceed the on-device pool and requires bit-exact greedy output with every offloaded block eventually fetched back. There is no throughput requirement; correctness is the bar.

    Guidelines and Hints¶

    Coding and debugging¶

    • Bit-exact grading means every step of your engine must agree with the reference. The most common cause of a mismatch is an off-by-one in the cache length or a wrong block-table index, which makes attention read one row too few, one too many, or an uninitialized slot. A useful check is to assert at each step that the cache length equals the prompt length plus the number of tokens generated so far.
    • Pass the current fill length to the attention call, not the allocated size, so attention never reads cache slots that have not been written yet.
    • Build the cache in stages. Get Part 1 bit-exact with a contiguous cache before introducing paging; a paging bug is much easier to find once you trust the contiguous path.

    Paging and the block table¶

    • With block size B, the block table entry at index i covers logical positions iB through (i+1)B - 1. Logical position p is in block ⌊p/B⌋ at offset p mod B. Allocate a new block exactly when the fill length crosses into a position the current table does not cover.
    • Test the allocator in isolation before wiring it into the engine: allocate a block and check the free count drops, free it and check the count returns. A paging bug that is really an allocator bug is much easier to see this way.

    Parts 3 and 4¶

    • For Part 3, a double-ended queue of free block indices makes a clean free list: allocation takes blocks from one end and freeing returns them to the other. The scheduler can hold pending requests in a similar queue and try to start the front request whenever a sequence finishes and frees memory.
    • Part 4 is deliberately open-ended. The interface is fixed; the eviction policy and how many blocks you move at a time are design choices. State them in the report and explain why you made them.

    Deliverables¶

    Submit through your course git repository as described in the introduction document. Include the engine files you edited (engine/engine.py, engine/kv_store.py, and the Part 3 and Part 4 files if you attempted them) and your report as report.pdf. Do not commit build artifacts.

    The grade is the bit-exact greedy-decode comparison against the in-folder reference engine. Throughput is not graded.

    Report¶

    Write about one page, single column, with tables where they help (captions required). Discuss the serving layer. Estimate how much work your cached engine saves over the recomputing reference at a sequence length of a hundred or so tokens, and say what the saving depends on (the context length and the model's width and depth). Discuss the paged cache: it produces the same tokens as the contiguous cache, as the bit-exact grading requires, so the question is what it costs and what it saves, and at what mix of sequence lengths paging earns back its indirection over simply reserving the maximum. If you completed Part 4, describe your eviction policy and the trade-off between how aggressively you offload and the fetch latency you pay when an offloaded block is needed.
