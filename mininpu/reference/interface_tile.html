    mininpu: Interface Tile¶

    The interface tile (npu/src/interface_tile/) is the device-memory controller: it serves off-chip device memory (PL-DDR4 on ZCU102) to the compute tile's DMU over port DMEM (dm_* req/ack) and, in the charter target, to the host-interface copy engine over port FABRIC (AXI4). The core module is dm_axi_bridge, which converts each dm_* load/store request to an 8-beat AXI4 INCR burst on m00_axi, the AXI4 master port connected to the DDR4 MIG and thus to off-chip PL-DDR4.

    The tile is a target-specific shell: the DDR / HBM controller changes per target (PL-DDR4 MIG on ZCU102, HBM on a future discrete card); it is not a compute unit and has no on-chip device-memory BRAM. In v1 the bridge runs on the same 50 MHz compute clock; the DDR/AXI side can be decoupled to the MIG user clock (200--300 MHz) with a CDC at the DMEM boundary, which is the first performance lever for [v1+]. The memory tile (an on-chip shared-L2 cache / staging tier, RFC #158) is [v3]-deferred and is absent here.

    Design SSOT: docs/charter.md §3, §5, §6, §13(B). Constants are owned by mininpu/isa / abi.py / params.py; this document does not restate them.

    Interface tile in context¶

    Slide summary (from the deck build)
    • Role: device-memory controller only (no host-plane DMA FSM, no on-chip device-memory BRAM).
    • DMEM: compute-tile DMA port (dm_* req/ack); [v1+] adds a CDC to run the AXI side at 200--300 MHz.
    • FABRIC: host-interface copy engine (AXI4); m00_axi drives PL-DDR4 via DDR4 MIG on ZCU102.

    Figure: if-context

    Interface tile in context. The dm_axi_bridge converts DMEM (dm_* req/ack from the compute tile's DMU) to AXI4 INCR bursts on m00_axi. FABRIC connects the host-interface copy engine (AXI4). A single m00_axi master drives PL-DDR4 via the DDR4 MIG. In [v1+] a CDC at the DMEM seam decouples the 50 MHz compute domain from the AXI/MIG clock (charter §6). The memory tile is [v3]-deferred and absent here.

    DMEM port. The compute tile exposes a BRAM-style DM interface (dm_addr, dm_din, dm_dout, dm_en, dm_we, dm_ack) that the dm_axi_bridge accepts on behalf of the interface tile. This is the sequencer's device-memory load/store path: every dmu.load.spm and dmu.store.spm instruction drives a dm_en request; dm_ack (combinational, 1 cycle wide at the DONE state) signals completion. In v1 the bridge runs on the same clk as the compute tile (50 MHz); in [v1+] a CDC (AXI clock converter or async FIFO) at this boundary enables the AXI/DDR side to run at the MIG user clock (200--300 MHz) independently (charter §6).

    FABRIC port. In the charter target the host-interface's copy engine (an AXI CDMA in v1) connects to the interface tile over a standard AXI4 port; the single m00_axi master is shared between DMEM and FABRIC through an arbitration mux (in npu.sv). In v1, DMEM and FABRIC use is sequential (host-mediated, depth-1); concurrent copy | compute is a [v2] item.

    Target-specific shell. The DDR4 MIG IP (ZCU102) and the m00_axi port sizing are board-specific; the dm_axi_bridge module is ASIC-portable. On ZCU102, m00_axi is a 32-bit AXI4 master at 50 MHz in v1.

    AXI4 INCR burst mechanism and bandwidth¶

    Slide summary (from the deck build)
    • One dm_* req → one AXI4 INCR burst: arlen/awlen=7 (8 beats × 4 B = 32 B; arsize=2; from dm_axi_bridge.sv).
    • [v1] 32-bit / 50 MHz = 200 MB/s; BDP ≈ 1 burst; single-outstanding async covers the latency.
    • [v1+]: widen to 256-bit; separate AXI clock (200--300 MHz) via CDC; multiple-outstanding + ID-tagged bridge.

    Figure: if-bw-clk

    Bandwidth budget and widening path. Left: as-built [v1] path (32-bit AXI at 50 MHz yields ≈200 MB/s, ≈1% of DDR peak). Right: [v1+] widened path (256-bit AXI at 200--300 MHz yields ≈9.6 GB/s); the CDC at DMEM enables the AXI/MIG clock to run independently of the 50 MHz compute tile, and ID-tagged slots map one-to-one onto dma_slot_done tokens.

    Burst FSM (from dm_axi_bridge.sv header). Load (dm_en=1, dm_we=0): IDLE → READ_AR → READ_R → DONE. Store (dm_en=1, dm_we=1): IDLE → WRITE_AW → WRITE_W → WRITE_B → DONE. In READ_AR the bridge asserts arvalid until arready; READ_R accepts rvalid beats into dm_dout one word per cycle, completing on rlast. In WRITE_AW/W the bridge streams wdata from dm_din (wlast on beat 7). DONE is one cycle wide: dm_ack is combinational off the state register, and the FSM returns to IDLE on the next rising edge.

    Burst parameters (hardcoded in dm_axi_bridge.sv): arlen/awlen = 7 (8 beats, AXI4 encoding), arsize/awsize = 2 (4 B per beat, 32-bit bus), arburst/awburst = 1 (INCR). One DM word is DM_DATA_WIDTH (256) bits = 32 B = 8 × 4 B, so one DM request maps to exactly one burst, no re-issue.

    Bandwidth in v1. The as-built path is 32-bit AXI at 50 MHz: 32,b × 50,MHz = 200,MB/s, roughly 1% of PL-DDR4 peak (~17 GB/s on ZCU102). At this bandwidth the bandwidth-delay product (BDP) is approximately one burst (200 MB/s × ~160 ns DDR latency ≈ 32 B), so bus width, not outstanding depth, is the binding bottleneck. Consequently [v1] = single-outstanding async: the sync DMA path blocks the sequencer FSM in-order; one async prefetch slot runs in the background while the sequencer computes on SPAD Port B (charter §5, §12).

    Widening path [v1+]. Widening the AXI data bus toward the 256-bit DM-word width and clocking the AXI/MIG side at 200--300 MHz grows throughput proportionally (256,b × 300,MHz ≈ 9.6,GB/s). Once the BDP grows large, multiple-outstanding transfers become worthwhile: the bridge carries per-slot AXI IDs (arid/awid, one per dma_slot_done slot), and a landing/reorder buffer sized to the BDP absorbs within-slot reordering (IHI 0022 § A5). The bridge itself remains ASIC-portable; SmartConnect and MIG adaptation stay board-side (charter §13(B)).

    Roadmap and deferred items¶

    All items below are charter-SSOT (charter §5, §10, §13(B)); the interface tile is not changed until the corresponding milestone.

    • [v1+] Widen AXI bus to 256 bit (DM-word width); add CDC at DMEM to run AXI/MIG clock at 200--300 MHz; multiple-outstanding with ID-tagged bridge and reorder buffer sized to the BDP.
    • [v2] Concurrent copy-engine | compute-tile access on m00_axi (FABRIC and DMEM in parallel); requires a real per-slot arbiter in place of the v1 sequential mux.
    • [v3] Replace DDR4 MIG with HBM multi-channel controller for the PCIe target; add the memory tile (on-chip shared-L2 cache / staging tier; RFC #158, deferred from this tile).

    Module summary¶

    File paths relative to npu/src/interface_tile/. The authoritative port list and parameter commentary live in each RTL file header. The AXI4 mux between dm_axi_bridge and the memory-tile AXI master lives in npu/src/npu/npu.sv.

    Table: interface tile module summary. The dm_axi_bridge is ASIC-portable; DDR4 MIG is board-level (ZCU102-specific).

    Module Source Role
    interface_tile interface_tile.sv Wrapper: exposes DMEM (ct_dm_* req/ack) and the AXI4 master port (FABRIC side); instantiates dm_axi_bridge; retains deprecated AXI-Stream paths (transition).
    dm_axi_bridge dm_axi_bridge.sv Converts dm_* req/ack to AXI4 INCR bursts on m00_axi: arlen=7, arsize=2, INCR; single-outstanding in v1; widened + ID-tagged in [v1+].
    DDR4 MIG (IP) Vivado IP catalogue PL-DDR4 controller (ZCU102-specific); user clock 200--300 MHz; instantiated at board level, not in the IP.
