    Verification and Simulation Models¶

    The RTL is verified with a cocotb testbench suite under dv/, driven per module (dv/**/test_<module>.py) and cross-cut by a coverage and sign-off plan. This page is the site-facing entry point for two things a first-time reader will otherwise not find a home for: the simulation models that stand in for hardware the testbench cannot instantiate, and a piece of simulation-only vocabulary (device_mem.sv, the tma_engine name) that names sim artifacts rather than shipping RTL. The methodology, coverage model, and sign-off criteria are owned by the dv/docs/ tree (dv/docs/verification.md, dv/docs/coverage.md, dv/docs/sign_off.md); this page does not restate them.

    Modelling device memory in simulation¶

    Off-chip device memory (PL-DDR4 on ZCU102, HBM2 on U280 later) is not part of the RTL under test: the interface tile drives it over AXI4 on m00_axi (see Uncore :: Device-memory controller). In simulation, device memory is modelled by a cocotbext-axi AxiRam instance attached to m00_axi in the interface-tile harness. That AxiRam is the v1 device-memory model; a lighter hand-rolled AXI4 RAM (sim/axi4_ram.py) is retained as a minimal stand-in for harnesses that do not need the full library.

    Simulation-only vocabulary (not hardware)¶

    Two names in the sources describe simulation or conceptual artifacts, not shipping RTL. They are recorded here so a reader who meets them does not mistake them for hardware modules.

    device_mem.sv (retired sim stub)¶

    device_mem.sv was a behavioral 256 KB SRAM stub (npu/src/device_mem.sv): the original cocotb stand-in for device memory. It is superseded in v1 simulation by the cocotbext-axi AxiRam on m00_axi described above, and the file is absent from the RTL tree on this branch. It is not a hardware component and never described one: it modelled a memory that lives off-chip. Treat any reference to it as a pointer to the retired sim model, not to a device unit.

    tma_engine: a conceptual name, and a retired module of the same name¶

    tma_engine is used two ways, and they are architecturally distinct:

    • Conceptual name (live). tma_engine is a conceptual name for the kernel-plane DMA unit inside the compute tile: the DMU. On this branch the DMU is a separate module that is currently stubbed (tile_sequencer exposes dma_slot_done / dmu_idle stubs; dmu.sv does not yet exist in the tree). It handles the sequencer-issued dmu.load.spm (0x40) / dmu.store.spm (0x42) instructions and is analogous to NVIDIA's TMA (cp.async.bulk.tensor). Its doctrine and ISA surface are on the DMU page.
    • Retired module. mem_tile/tma_engine.sv was a synchronous 1D word-by-word DMA between device memory and the memory tile; it was dissolved together with the memory tile (PR #62). The conceptual name above reuses the string, but the retired module and the DMU are not the same design. The memory tile itself is [v3]-deferred (RFC #158).

    Numerical tolerance¶

    L4/L5 tests compare RTL output against the ISS golden with a per-opcode absolute tolerance, consolidated here:

    Op class atol
    Elementwise ALU ops (add / sub / mul / scale) 1e-4
    Special functions (exp2, rsqrt, gelu) and the end-to-end GPT-2 forward-pass output 1e-3

    The looser 1e-3 bound reflects the BRAM-LUT approximation error of the transcendental units (kept within about 0.1%, see VPU :: Why BRAM-LUT non-linear units?).

    v1 acceptance (the worked example)¶

    The +1-vector kernel is the v1 acceptance worked example: memcpy(H2D) -> launch -> memcpy(D2H), asserting read_devmem == input + 1 element-wise. Its coverage set is the minimal end-to-end ABI/DMU surface:

    • the doorbell self-clear;
    • instr_ready completion plus the -EBUSY backpressure return;
    • the in-kernel dm_ack load/store completion;
    • the make-async-real dma_slot_done set/clear plus the flush.slot fence (the functional test that was previously missing).

    The ABI-snapshot gate stays green except for the one planned ABI_VERSION 7 -> 8 cutover.
