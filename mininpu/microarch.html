This document describes the compute-tile microarchitecture behind ISA 0.5.0. Content is split into a Locked design and a Decisions pending section (still-open items).

Version context: ISA 0.5.0 is the design target; the shipped SSOT (opcodes.py) is still 0.4.0 and the cutover (0.5.0 opcodes, PE_LATENCY 0 -> 2) is pending. See the ISA 0.5.0 status note.

Concurrency model

In-order issue, 1 instruction/cycle. No producer-busy scoreboard; no hardware interlock. All hazard avoidance is static compiler scheduling. flush.slot is the only mid-program runtime synchronization (halt is the terminal drain at end of kernel).

Nonblocking set = {mxu.matmul, mxu.load.w, mxu.acc.init, async-DMA}; everything else is blocking.

Distributed per-unit AGU

Each unit has its own address-generation unit. The AGU is an adder: effective address = operand_literal + bo[channel], no multiplier.

operand_literal is a compile-baked immediate or pool index (devmem bases GRID_LINEAR launch-patched, not a CSR); bo is the per-channel base-offset broadcast from the tile sequencer.

Base-offset (bo) addressing

8 base-offset channels in the sequencer; the AGU adds literal + bo with no multiplier.

bo is maintained by pool-sourced stride vectors: LIFO push/pop at loop.begin / loop exit, and bo[ch] += stride[ch] at each loop.end. Full rule: the ISA addressing model.

Literal pool (LPM)

On-chip param RAM in the sequencer (256 x 64b (~2 KiB), provisional; final sizing during implementation), pool-index space, baked + launch-patched, read in Decode.

Realization: reuse the mature IRAM-embedded literal-pool machinery (loaded with the kernel image, read in Decode via the redirect mux), not a new separate RAM (see the implementation note).

Holds loop descriptors (hi + stride vector), devmem bases, and scalar immediates. See the ISA literal-pool section.

02 Execution model / pipeline

Locked - the fully-exposed, compiler-timed pipeline.

Performance model · throughput-bound, not latency-bound

The compute tile is throughput-bound. Per-instruction latency - including D_vpu and the matmul systolic fill - is negligible for performance: it is the pipeline drain latency, amortized across the tile-loop, not on a serial critical path. So pipeline depth may grow freely to meet timing while throughput stays 1 tile-row/cycle (VPU) and 1 column/cycle (MXU).

This rests on row-pipelining, which is partial. The single-source SIMD ops (scale.bcast, scale.col.bcast, scale.imm, sub.bcast, add.col.bcast) stream at 1 tile-row/cycle with multiple rows in flight (rows overlapped); for them D_vpu is only the drain latency of a single row, paid once and hidden behind the following rows, hence negligible. gelu / exp2 and the two-source ops stay latency-serial (each row drains the full D_vpu before the next issues). softmax / LayerNorm are dependent reduction chains within a row, but the tile-loop supplies the row-level parallelism the pipeline consumes, so for the streamable ops the pipe stays full.

The level-wise reduce-tree pipelining and the deep fp pipelines are intended design, not overhead: fp arithmetic follows standard pipelined designs (FloPoCo-style; fp_mul combinational, IEEE fp_add registered / multi-stage), so deepening a stage to close 200 MHz spends latency (which does not matter) to preserve throughput (which does).

  • Stages: F (fetch IRAM[PC], 1/cy) to D (decode + per-unit AGU address gen + iv broadcast) to X (execute, op-specific latency) to W (writeback to VREG / ACC / SPAD / DRAM per the op's writeback target). This is the architectural issue model; the physical Fetch is a pipelined multi-cycle IRAM BRAM read (~3 cycles) that still sustains 1 instruction/cycle issue in steady state (see the Doc 3 tile-sequencer FSM).
  • Fully exposed pipeline: the sequencer issues exactly 1 instruction per cycle and NEVER stalls to wait on a result. The only runtime waits are flush.slot (the sole mid-program sync, waits on an async-DMA slot) and halt (the terminal drain at end of kernel, not a mid-program sync).
  • Fixed, data-independent latencies: the compiler covers every latency by placing dependents later and filling gaps with independent instructions or NOPs. No hardware interlock, no scoreboard.
  • Nonblocking ops (mxu.matmul, mxu.load.w, mxu.acc.init, async-DMA) run in background FUs: the sequencer issues them in 1 cycle and proceeds. The compiler overlaps them (e.g. a background matmul concurrent with a foreground VPU op) and double-buffers the shared VREG tiles.
  • Completion: compute nonblocking ops are fixed-latency, so they need NO fence (compiler-timed). async-DMA is variable-latency, so the compiler inserts flush.slot before any consumer of the DMA'd region.
  • Structural single-instance limits: 1 MXU (matmuls serialize), 1 inactive weight buffer (at most 1 load.w preload ahead), 2 outstanding DMA slots (DMA_SLOT_COUNT=2; flush.slot's bitmask selects among them, async-only).
  • Latencies: the structurally-fixed values are N and the systolic fill = 2N-1; the VPU pipeline depth D_vpu, the registered-PE MAC / PE pipeline depth, and the absolute offsets (d, L_matmul) are RTL / synth-timing measured implementation numbers (D_vpu currently about 23 (measured after the fp-pipeline pass)), NOT ISA-visible structural constants.

03 Functional units and memories

Locked - unit geometry and roles.

MXU

8x8 weight-stationary systolic array, bf16 x bf16 to fp32 accumulate, 2 weight buffers (active / inactive double-buffer).

X streams from VREG, cast fp32 to bf16 at the PE input (no separate X input buffer). Output to ACC. Weight-stationary dataflow.

PE MAC is registered for 200 MHz (PE latency to be confirmed by synth); systolic fill = 2N-1 plus the PE pipeline depth.

VPU

Fixed-depth pipeline, 8 lanes, function-select {reduce | lut | simd}. Lane = column, row-serial: one 256-bit tile-row per cycle, 8 columns in parallel (the natural structure for the per-row reductions in softmax / LayerNorm). Throughput = 1 tile-row/cycle (row-pipelined, multiple rows in flight) for the single-source SIMD ops; gelu / exp2 / two-source ops are latency-serial (1 row / D_vpu).

Pipeline depth D_vpu is RTL / synth-measured (currently about 23 (measured after the fp-pipeline pass)), an implementation number, NOT an ISA-visible structural constant. fp_mul stays combinational (maps to a DSP), but an IEEE fp_add is registered (1+ pipeline stages, standard FloPoCo-style: a single combinational IEEE fp_add is ~11 ns and misses the 5 ns / 200 MHz budget); the reduce-sum tree is level-wise pipelined and each BRAM-read to compute cone is registered. D_vpu is therefore the sum of registered stages (RTL-measured), not an fp-op count. The deepest function (gelu) sets it; the other functions (reduce tree, exp2 BRAM, rsqrt, simd) align to the same depth so the reservation window is single. Reuses the fpu/* primitives (exp2 / rsqrt / tanh / recip BRAM LUTs, fp_mul / fp_add).

VREG

32 x [8x8] fp32 = 8 KiB. Whole-tile-row access (256-bit), 2 read + 1 write ports (2r1w baseline).

No banking: MXU / VPU / LSU conflicts are avoided by temporal separation (foreground VREG access scheduled outside the short MXU X-read burst). See the realization subsection.

ACC

[8x8] fp32 x depth 2 = 512 B. 8 per-column banks (the physical column organization: one accumulator column per systolic column). Separately, ACC buffers {0,1} are both used for depth-2 double-buffering; the ACC buffer field (the depth index) selects the buffer, not a column bank.

Depth-2 double-buffer: the second buffer overlaps a block's drain/store tail (acc.store → gelu → vreg.store) with the next block's matmul.

SPAD

bf16 storage. Separate, 0-based on-chip address space (like CUDA .shared). 32 KiB, 8 banks.

LSU (load/store unit)

A small dedicated load/store unit handling acc.store (ACC to VREG, fp32), vreg.load (SPAD to VREG, bf16 to fp32), and vreg.store (VREG to SPAD, fp32 to bf16).

Loop control

loop.end branch bubble = 0: a counted loop with a precomputed target.

Full-unroll is infeasible (the unrolled GPT-2 static instruction count far exceeds the 4096-deep IRAM; a wider N=16 systolic array - the same systolic tile dimension N, a microarch sweep target, not an independent outer factor - gives about 8x fewer tiles than N=8), so outer tiling must be in-kernel nested loops driven by the base-offset (bo) accumulators (full rule: the ISA addressing model).

Conflict model

No runtime interlock or scoreboard: the compiler statically avoids all VREG / SPAD port and bank conflicts using the reservation table (see the ISA timing contract); sim == synth is mandatory.

Optional debug-only conflict detector flags same-port / same-bank same-cycle access in debug builds (no stall): an optional bring-up aid, not part of the shipping datapath.

VPU pipeline

Locked structure - fixed-depth, partially row-pipelined vector pipeline (single-source SIMD ops stream; gelu / exp2 / two-source stay latency-serial); the depth D_vpu is RTL / synth-measured (currently about 23, measured after the fp-pipeline pass), not a locked constant.

  • Structure: 8 lanes (lane = column), row-serial at one 256-bit tile-row per cycle, throughput 1 tile-row/cycle (row-pipelined: multiple tile-rows in flight, 1 issued per cycle, each draining after D_vpu cycles) for the single-source SIMD ops only (scale.bcast, scale.col.bcast, scale.imm, sub.bcast, add.col.bcast); gelu / exp2 / two-source ops are latency-serial (1 row / D_vpu). fp_mul stays combinational (DSP), but an IEEE fp_add is registered (1+ pipeline stages, standard FloPoCo-style: a single combinational IEEE fp_add is ~11 ns and misses the 5 ns / 200 MHz budget); the reduce-sum tree is level-wise pipelined and each BRAM-read to compute cone is registered.
  • Depth is RTL-measured, set by gelu: D_vpu is the sum of the registered stages along the deepest function (gelu: fp_mul / registered fp_add chain + the tanh BRAM lookup), measured from RTL / synth (currently about 23, measured after the fp-pipeline pass), not a hand-counted fp-op total; the other functions (reduce tree, exp2 BRAM, rsqrt, simd) align to the same depth so the reservation window is single. D_vpu is an implementation number, in the same bucket as L_matmul and the PE pipeline depth, NOT an ISA-visible structural constant (unlike N and 2N-1).

VREG realization (whole-tile-row RF + MXU-burst temporal separation)

Locked - resolved model.

  • Physical model: VREG = 32 registers, each an [8x8] fp32 tile (the paper design lifts the current 1D 8-wide vector RF to 2D tiles; hwparams VREG_SIZE 8 -> 64). Access granularity is a whole tile-row (256-bit).
  • Ports: 2 read + 1 write (whole tile-row), matching the whole-tile-row 2r1w RF. A 2nd write port is a sizing option only if two writes ever need to be concurrent; tail sub-ops are dependency-serial, so 2r1w is the baseline.
  • MXU X read: the MXU reads its X tile from VREG in a short burst (~N cycles) into its x_matrix staging buffer, then does not touch VREG for the rest of the matmul.
  • Conflict avoidance = temporal separation, not banking: the compiler schedules the foreground unit's (VPU / LSU) VREG accesses outside the MXU X-read burst window. Both windows are short, so this is easily schedulable.
Compiler rule

Keep the MXU X-read burst and any concurrent VPU / LSU VREG access in disjoint cycles. The MXU burst is a small fraction of the matmul, so the foreground unit has VREG to itself otherwise. Tail ordering is handled by the scheduler. The VPU has a single fixed pipeline depth (D_vpu, RTL / synth-measured, uniform across functions), so a VPU result is available at issue + D_vpu.

SPAD access model

Locked.

  • Geometry: 32 KiB = 1024 x 256b rows, 8 banks (bank = row mod 8), bf16 storage.
  • Clients: MXU load.w (weight read) and mxu.acc.init (bias read); LSU (vreg.load read, vreg.store write); DMU (devmem <-> SPAD, async). X is now sourced from VREG, so the MXU no longer streams X from SPAD. The VPU does not access SPAD directly - it operates on VREG; SPAD <-> VREG moves go through the LSU.
  • Fixed-latency contract: compute clients (MXU load.w / acc.init, LSU) get a fixed SPAD access latency; bank conflicts among compute clients are compiler-avoided (static scheduling, like VREG).
  • DMU absorbs the arbitration variability: on a bank conflict the DMU yields (it is async and flush-gated, so its latency is already variable / tolerated), so compute never stalls on SPAD. This reuses the existing banked_sram arbiter with priority compute (fixed) > DMU (yields).
Note

SPAD row access latency is RTL-derived (MEM_LATENCY=2 -> about 3 cycles per row) and enters the reservation table as a fixed value; the end-to-end reservation offsets remain TBD.

04 Overlap and roofline

Locked - qualitative bounds only.

  • MXU || VPU overlap: the compiler runs a background nonblocking op (e.g. a matmul) concurrent with a foreground blocking op (e.g. a VPU op) and double-buffers the shared VREG tiles.
  • MLP / matmul-heavy blocks are MXU-bound.
  • Tiny devmem matmuls are DRAM-BW-bound (bounded by the HP-port DRAM bandwidth, not on-chip compute).

05 Decisions pending

Still-open items.

Pending · reservation-window offsets

The reservation offsets (d, L_matmul, the MXU PE pipeline depth, the VPU pipeline depth D_vpu, and the absolute numbers in the ISA timing contract) are pending RTL / synth-timing and must be sim == synth verified. D_vpu is an implementation number (currently about 23 (measured after the fp-pipeline pass)), in the same bucket as L_matmul and the PE pipeline depth; only N and the systolic fill 2N-1 are the ISA-visible structurally-fixed values.

Pending · DMU internals

Descriptor slots, burst sizing, dm_axi error / timeout handling, and arbitration are deferred to implementation.

Pending · final sizing

LPM size and base-offset channel count final sizing. The VREG realization is now resolved (whole-tile-row RF + MXU-burst temporal separation; see the locked Functional units section). A 2nd VREG write port remains a sizing option only.
