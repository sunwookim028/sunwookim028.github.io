    TikZ figures: source + SVG¶

    The 40 SVGs in this directory are the mkdocs-embeddable form of the diagrams that were drawn in TikZ. They are embedded by the topic pages under ../../reference/ and indexed by ../index.md (the figure gallery).

    The .tex next to each .svg is the source of truth. Each NAME.svg is a build product of the NAME.tex beside it, rendered through the recipe in How they are generated below. Edit the TikZ and re-render; do not hand-edit the SVG, or the next re-render silently reverts you.

    mininpu-tikz.sty is the shared style (role colours, block styles, \npusep) that all 40 sources \usepackage. It is the only dependency they have: every figure is \documentclass[tikz,border=4pt]{standalone} + \usepackage{mininpu-tikz}.

    History. The corpus briefly lived as SVG-only: the sources were at docs/src/figures/*.tex and were deleted wholesale when docs/ became mkdocs-only, leaving 40 outlined SVGs that could not be edited or regenerated. They were recovered from git history and re-homed here, beside the SVG each one produces. The .tex files here are the sole general exception to the "no LaTeX in docs/" rule in CLAUDE.md: they are figure source, never built by mkdocs and never published.

    Two further points, both deliberate:

    • Six figures are duplicated into labs/common/figures/ (arch-l1-system, vpu-datapath, vpu-reduce, ct-pingpong, mxu-pe, mxu-array), together with a second copy of mininpu-tikz.sty. The labs/ handouts are still LaTeX and cannot include SVG, so those six are vendored there as source + PDF. The two .tex copies are byte-identical today, but nothing enforces that: if you change one of those six, update all three artifacts (the .tex here, the .tex + .pdf in labs/common/figures/, and the .svg here). See Open: the six-figure duplication.
    • Four figures are not referenced by any topic page (if-addrmap, if-devmem, if-dma-fsm, if-worked). They were built and committed in the LaTeX era but never included. They are embedded in the figure gallery rather than dropped, so removing them stays a decision rather than an accident.

    How they are generated¶

    The standalone document class crops via pdfTeX page primitives, which do not exist in DVI mode, so dvisvgm's page box lands on the TeX reference point and carries dead space. The build therefore converts via DVI (for correct glyphs) and then crops to the true ink bounding box.

    There is no Makefile for this: figures change rarely, and the recipe is three commands. Run it from this directory. Requires latex + dvisvgm (TeX Live) and python3 with cairosvg + pillow (both are in the mininpu conda env).

    # 1. TikZ -> DVI  (pgfsys-dvisvgm.def makes pgf emit dvisvgm specials)
export TEXINPUTS=".:$PWD:"
latex -interaction=nonstopmode -output-directory=/tmp/fig \
      "\def\pgfsysdriver{pgfsys-dvisvgm.def}\input{NAME.tex}"

# 2. DVI -> SVG  (outlined glyphs, tight bbox)
dvisvgm --no-fonts --exact -o /tmp/fig/NAME.svg /tmp/fig/NAME.dvi

# 3. crop the viewBox to the rasterized ink bbox + 4pt (matches border=4pt),
#    then install over the committed SVG
python3 crop_viewbox.py /tmp/fig/NAME.svg && cp /tmp/fig/NAME.svg NAME.svg

    crop_viewbox.py is committed beside the figures it crops: it is step 3 of this recipe, so it is source like the .tex files are, not prose to copy-paste. It takes one argument (the SVG to crop, rewritten in place), has no options, and reads its PAD = 4.0 / SCALE = 8 constants from the top of the file. Edit it there rather than in this README.

    What "reproducible" means here¶

    Re-rendering is not byte-identical to the committed SVG, for two benign reasons. Both were confirmed by re-rendering mxu-pe, dm-flow, ct-pingpong and seq-lifecycle against their committed SVGs:

    • <defs> glyph order. dvisvgm emits the glyph <path> defs in an order that depends on its font cache state (a one-shot run orders them differently from a warm batch run). The set of lines is identical: dropping the <svg …> header and sorting the remaining lines gives an identical md5 for all four figures, and the ids are referenced by <use>, so order carries no meaning.
    • Crop quantization. Step 3 measures the ink bbox by rasterizing, so the viewBox depends on SCALE. At SCALE=8 it lands within ~0.2pt of the committed values. Rendering both under the same header is pixel-exact (0 differing pixels of ~1.9M at 1600px wide), so the residual is the crop measurement, not content.

    So: verify a re-render by pixel-comparing, not by cmp. A re-render whose sorted body matches and whose viewBox is within a fraction of a point is correct.

    Decisions worth not re-litigating¶

    • DVI route, not PDF route. dvisvgm --pdf on a pdflatex-built PDF crops correctly but silently drops the \npusep middot glyphs (dm-flow: 637 vs 648 glyph <use> elements). The DVI route keeps every glyph.
    • No svgo. It cuts 3.7M to 2.2M but drops group stroke colour declarations on some figures (mxu-pe #5a5a5a, if-devmem #d55e00). The size win is not worth unverifiable colour changes.

    Open: --no-fonts costs searchable text¶

    All 40 committed SVGs have a <text> count of 0: --no-fonts outlines every glyph into <path> + <use>. Figure labels are therefore not selectable, not greppable, and invisible to screen readers. This is a real accessibility and searchability cost, and now that the sources are back it is a live choice rather than a fait accompli.

    Measured on mxu-pe and seq-lifecycle (dvisvgm 3.2.1):

    figure --no-fonts (committed) --font-format=woff2
    mxu-pe <text> 0, <use> 300, 75065 B <text> 56, <use> 0, 28676 B (-62%)
    seq-lifecycle <text> 0, <use> 303, 76483 B <text> 46, <use> 0, 27135 B (-65%)

    The fonts are embedded, not referenced. The woff2 output carries 8 (mxu-pe) and 5 (seq-lifecycle) @font-face blocks, each with a src:url(data:application /x-font-woff2;base64,…) payload and zero external url(http…) references. So "the CM fonts are not installed on the reader's machine" is not a failure mode here: the SVG stays self-contained.

    The actual failure mode is renderer text layout. Rendered through cairosvg (the only SVG renderer installed here), the woff2 variant is legible but visibly wrong: inter-word spacing collapses (weight in -> weightin, activation in -> activationin), subscripts misplace, and bold weight is lost. dvisvgm positions each text run at explicit x/y and relies on the renderer honouring the embedded font's advance widths; cairosvg does not, so the layout drifts. Outlined text has no layout to get wrong and renders identically in every engine.

    This does not mean woff2 is broken in browsers, which is where the docs are actually read. It is the same gap the original conversion flagged and it is still open: no browser and no second renderer (rsvg-convert, resvg, chromium, firefox) is installed in this environment, so "woff2 renders correctly in a real browser" remains unverified in both directions. The tradeoff:

    • --no-fonts (status quo): renders identically everywhere; 2.7x larger; text is not selectable, greppable, or screen-readable.
    • --font-format=woff2: 2.7x smaller, real <text>, self-contained fonts; correctness depends on the renderer honouring embedded-font metrics, confirmed wrong under cairosvg and unconfirmed under any browser.

    To settle it, render one figure both ways and open both in Chrome and Firefox. If woff2 is clean in both, switching is a strict win and worth doing for all 40.

    Open: the six-figure duplication¶

    arch-l1-system, vpu-datapath, vpu-reduce, ct-pingpong, mxu-pe and mxu-array now exist as .tex in two places (here and labs/common/figures/), as does mininpu-tikz.sty (7 duplicated files, currently byte-identical). labs/ needs LaTeX+PDF; docs/ needs SVG; nothing keeps the two .tex copies in sync, so the next edit to one of the six can silently skew the handout from the docs page. Options, for the owner to pick:

    1. Canonical here, symlink from labs/. labs/common/figures/NAME.tex -> ../../../docs/mkdocs/figures/tikz/NAME.tex. Zero build changes; relies on symlinks surviving checkout (fine on Linux/macOS CI, breaks on Windows and in some archive exports).
    2. Canonical here, copy at build time. A labs/ Makefile prerequisite copies the six .tex + the .sty before latex runs. Robust and explicit; adds a generated-file rule and a .gitignore entry to labs/.
    3. Canonical here, TEXINPUTS instead of copying. Point the labs/ build at this directory via TEXINPUTS and delete the vendored .tex. No copies, no symlinks; couples the labs/ build path to a docs/ directory layout.
    4. Third location (figures/ at repo root) that both consume. Cleanest conceptually, largest diff, and moves 40 files for the sake of 6.
    5. Keep the duplication, add a CI drift gate that diffs the two copies and fails on skew. Smallest change; keeps the duplication but makes it loud.

    A drift gate (5) is worth having under any of 1-4 that leaves two copies, and is the cheapest immediate protection. This note is a proposal only: no consolidation has been done, and labs/common/figures/ is untouched.
