1. Overview

The mininpu compute tile is a matmul/vector accelerator exposing a small, statically-scheduled instruction set of 15 opcodes. The programmer-visible pipeline issues in-order, single issue, at exactly one instruction per cycle. There is deliberately no hardware interlock and no producer-busy scoreboard: the hardware never stalls a consumer waiting for an in-flight producer, and it never detects a write that clobbers an operand still being read. All hazard avoidance, latency padding, and structural-resource serialization are the responsibility of the compiler's static schedule (Section 6).

Every instruction is a 64-bit fixed-width word. Operand storage is fixed by opcode; there is no runtime type tag in the encoding (Section 3).

Data types follow a fixed convention across the memory spaces: bf16 in SPAD and DRAM (devmem); fp32 in VREG and ACC. The only dtype-converting instructions are vreg.load (bf16 -> fp32 widen on SPAD -> VREG) and vreg.store (fp32 -> bf16 truncate on VREG -> SPAD). The MXU casts its streamed X operand fp32 -> bf16 at the PE input; acc.store is an exact, lossless fp32 copy ACC -> VREG.

Launch model: a kernel is a binary of NPU assembly. At launch, the command processor loads the kernel binary into IRAM[0] and starts the sequencer, which runs to the terminal halt instruction. halt drains all outstanding DMU transfers, then asserts done. Because there is no hardware reorder, speculation, or interlock, program correctness is provable from the instruction stream alone given the fixed, data-independent instruction latencies (Section 5); a binary is only valid for the microarchitectural latencies it was compiled against.

2. Architectural State and Geometry

The tile exposes four explicitly-addressed data spaces (analogous to CUDA .global / .shared / .reg plus a tensor accumulator), plus MXU-internal weight buffers and control state. There is no unified / generic addressing: each space has its own address form and the space is implied by the opcode/operand slot.

spaceroleaddressed bygeometry
devmem off-tile DRAM (host-shared, global) byte address via literal-pool index + D4 launch patch 4 GB (DM_ADDR_WIDTH=27, 256b words)
SPAD on-chip scratchpad (shared, explicitly managed, NOT a cache) SPAD row index (10-bit) 1024 x 256b = 32 KiB, 8 banks (addr % 8)

wbuf weight buffer (MXU, stationary) implicit (no operand); mxu.load.w fills inactive, mxu.matmul promotes 2 (double buffer)
control PC + induction vars - PC 12-bit; iv_reg x4 (LIFO nest depth 4), 32-bit each

The datapath spine:

devmem  <--DMU-->  SPAD  <--cast-->  VREG  <-->  ACC
  • W path: SPAD -> wbuf via mxu.load.w (weights are stationary).
  • X path: VREG -> PE array, streamed (row-stationary hybrid), cast fp32 -> bf16 at the PE input.
  • Output path: PE array -> ACC.

The dtype boundary sits at the SPAD <-> VREG edge: bf16 lives in SPAD/DRAM, fp32 lives in VREG/ACC, and the conversion happens only on vreg.load / vreg.store.

3. Instruction Encoding

Every instruction is a single 64-bit word with a fixed field layout:

 63        56 55      48 47        32 31        16 15         0
+-----------+----------+------------+------------+------------+
|  OP[7:0]  | FLAGS[7:0]|  op0[15:0] |  op1[15:0] |  op2[15:0] |
+-----------+----------+------------+------------+------------+
  • OP[63:56] - 8-bit opcode.
  • FLAGS[55:48] - 8-bit flags byte (per-op meaning; unused bits are reserved-must-be-zero).
  • op0[47:32], op1[31:16], op2[15:0] - three 16-bit operand slots.

Operand storage (which slot holds a SPAD row vs. a VREG id vs. a devmem pool-index, etc.) is fixed per opcode; there is no type tag in the word.

3.2 Per-instruction semantics

0x00 mxu.load.w - Load a weight tile from SPAD row op0 into the inactive weight buffer (the MXU keeps two, double-buffered). Weights are stationary; the tile becomes active when the next mxu.matmul promotes it. Flag [0]=transB reverses the weight-row load order (transposed-B load). Each mxu.matmul consumes exactly one preceding mxu.load.w.

0x01 mxu.matmul - Streaming matmul into ACC bank op0 using the X operand from VREG id op1 and the currently-active stationary weights:

acc_out = (accum ? acc_out : 0) + X_streamed @ W_active^T

X is read from VREG (fp32) and cast fp32 -> bf16 at the PE input; W is the stationary tile from the most recent mxu.load.w. Flag [0]=accum selects accumulate-into vs. overwrite; flag [1]=causal masks the upper triangle of the output (attention causal mask). A matmul with no pending mxu.load.w is illegal.

0x02 mxu.acc.init - Seed ACC bank op0 with a bias vector read from SPAD row op1 (a 1 x N row), broadcast to all rows of the accumulator tile. Used to initialize a K-accumulation with a bias instead of an accum=0 first matmul.

0x10 vpu.reduce - Per-row reduction of VREG src op1 into VREG dst op0, writing the per-row scalar result into column 0. Flag [0] selects sum (0) or max (1).

0x11 vpu.lut - Elementwise unary LUT/approx over VREG src op1 into VREG dst op0. Flag [1:0] selects the function: 0 = exp (base-e), 1 = rsqrt, 2 = gelu.

0x12 vpu.simd - Elementwise SIMD ALU op. op0 = dst, op1 = operand a, op2 = operand b (or a literal-pool immediate when bcast=imm). Flag [4:2]=funct selects {add, sub, mul, max, fma, mov}; flag [1:0]=bcast selects the broadcast mode {none, row, col, imm} for the b operand.

0x30 acc.store - Drain ACC bank op1 into VREG dst op0. This is an exact, lossless fp32 copy (ACC -> VREG); no dtype conversion.

0x32 vreg.load - Load SPAD row op1 (bf16) into VREG dst op0, widening bf16 -> fp32. This is one of the two dtype-converting instructions.

0x33 vreg.store - Store VREG src op1 (fp32) into SPAD row op0, truncating fp32 -> bf16. The other dtype-converting instruction.

0x40 dmu.load.spm - Asynchronous DRAM -> SPAD copy. op0 = destination SPAD row; op1 = devmem source address as a literal-pool index (the DRAM stride is read from pool[idx+1]); op2 = shape packed as (rows-1)<<8 | (cols-1). Flag [4]=IMM_OP1 marks op1 as immediate. Single-outstanding; the region is not safe to consume until a matching flush.slot.

0x42 dmu.store.spm - Asynchronous SPAD -> DRAM copy. op0 = source SPAD row; op1 = devmem destination (pool index); op2 = shape (0xFFFF means derive the shape at lowering). Flag [4]=IMM_OP1 as above.

0x50 loop.begin - Open a loop frame on induction variable op0. op1 = hi bound (literal-pool index); op2 packs (lo<<8) | step as two 8-bit inline literals. Pushes a loop frame (LIFO, nest depth 4) and sets IV = lo. Flag [4]=IMM_OP1.

0x51 loop.end - Close the loop frame for induction variable op0: IV += step; branch back to the loop body if IV < hi, else fall through and pop the frame.

0x53 flush.slot - Region fence for asynchronous DMU. op0 is a 16-bit slot bitmask; the instruction blocks until the indicated DMU slot(s) report done. This is the only runtime synchronization primitive.

0x5F halt - Terminal instruction. Drains all outstanding DMU transfers, then asserts done. Ends kernel execution.

4. Flags Summary and LOCKED Decisions

4.1 Locked decisions

  • flags[5] = ACC_TILE is DROPPED from vpu.simd. The VPU is VREG-only. Bit 5 is reserved-must-be-zero on all opcodes.
  • ACC bank addressing is implicit. acc_out / acc_src are in [0, ACC_DEPTH=2); an index >= depth is an illegal-op. An explicit bank operand is deferred to issue #313 / the depth sweep #262. v1 uses bank 0 only.
  • VREG is a 2D [8x8] fp32 tile. The encoding is unchanged (5-bit id); the register is a tile, not a flat vector.

4.2 Reserved-must-be-zero

  • All unused flag bits on every opcode.
  • flags[5] on all opcodes (dropped ACC_TILE).
  • DMU flags[5] (the async-mode flag is retired - async is the only mode).

6. Compiler Scheduling Contract

Because the tile issues in-order, one instruction per cycle, with NO hardware interlock and NO scoreboard, the correctness of a program depends entirely on the compiler's static schedule. Program correctness must be provable from the instruction stream alone, independent of microarchitectural timing (given the fixed latencies of Section 5). The compiler MUST obey every rule below.

6.1 Latency-respecting placement (RAW)

Schedule any consumer of a result no earlier than:

consumer_issue_cycle >= producer_issue_cycle + producer_latency

using the fixed latencies from Section 5. Fill the gap with independent instructions or NOPs. A consumer issued too early reads stale data - there is no hardware stall to save it.

6.2 WAR on inputs is unguarded

Do not overwrite a register / SPAD row / ACC bank that an in-flight instruction is still reading. In particular:

  • Do not write a VREG that an in-flight mxu.matmul is still streaming as its X operand.
  • Do not overwrite a SPAD weight tile whose mxu.load.w has not finished.
  • Do not overwrite matmul input rows while they are being consumed.

The hardware does not detect any of these anti-dependences; the compiler must keep the write out of the producer's read window.

6.3 WAW ordering

Two writes to the same destination must be program-ordered and must not have overlapping write windows on the same location. The compiler orders them and spaces them so the later write lands strictly after the earlier one completes.

6.4 Structural hazards (one physical unit)

Each functional unit is a single physical resource; the compiler must serialize uses of the same unit:

  • MXU: executes one matmul at a time. Matmuls into the same ACC serialize. At most one mxu.load.w preload is in flight (single inactive weight buffer). Each matmul consumes exactly one preceding mxu.load.w - a matmul with no pending load.w is ILLEGAL.
  • VPU: runs one op-stream at a time. The next VPU op's reads may begin only after the current op's read window closes.
  • Throughput: respect per-op throughput (matmul 8 / K-tile, VPU 8 / op).

6.5 VREG bank-conflict-free scheduling (2r2w, banked per-column)

VREG exposes two read ports (R_mxu = MXU X-stream, R_vpu = VPU lane) and two write ports (W_mxu = acc.store, W_vpu = VPU result / vreg.load cast). It is realized as banked single-ported SRAMs (bank = function of reg-id), NOT a true multiport. The compiler must schedule so that:

  • the two concurrent reads land in different banks, and
  • the two concurrent writes land in different banks

with no same-bank same-cycle conflict. This conflict-free banking is exactly what makes the MXU || VPU one-tile software pipeline work.

6.6 DMU / memory ordering via flush.slot

DMU transfers are asynchronous and single-outstanding (v1). Ordering rules:

  • Before any instruction consumes data that a dmu.load.spm brought into a SPAD region (a vreg.load or mxu.load.w reading that region), insert a flush.slot for that DMA slot.
  • Before a dmu.store.spm reads a SPAD region, ensure the producing vreg.store writes have completed.
  • halt drains all outstanding DMU.
  • Two asynchronous DMAs to the same region must be fenced against each other.

6.7 ACC discipline

For a K-accumulation:

  1. Issue mxu.acc.init (or an accum=0 first matmul) to seed the ACC.
  2. Issue the accumulating matmuls (accum=1) into the same ACC bank, in order.
  3. Issue acc.store to drain the result to VREG.

ACC depth 2 permits double-buffering (drain tile j while tile j+1's matmul fills the other bank), but only two accumulator tiles are ever resident.

6.8 Loops

Trip counts are literal-only: lo and step are 8-bit (0..255) inline literals; hi is a literal-pool value. There is no runtime-data-derived trip count. The iv_reg nest depth is 4 (LIFO).

6.9 Addressing

All operand addresses are compile-baked:

  • SPAD rows / VREG ids / ACC banks are literals.
  • devmem addresses are literal-pool indices patched at launch (D4 GRID_LINEAR + pool_base).
  • IV-relative addressing (base + iv*stride) uses the sequencer-broadcast induction variable.

There is NO runtime data-dependent addressing in v1.

6.10 The MXU || VPU software pipeline

To overlap execution (attention, mixed blocks), the compiler:

  1. Issues the background nonblocking ops (mxu.matmul, mxu.load.w, mxu.acc.init, async DMA), then
  2. Issues the foreground blocking op (VPU / acc.store) so their execution windows overlap, and
  3. Double-buffers the shared VREG tiles so producer and consumer use disjoint registers / banks (see 6.5).

For matmul-heavy blocks (MLP), the VPU is far under-subscribed, so no overlap is needed.

6.11 Determinism / recompilation

There is no hardware reorder, speculation, or interlock. A binary is valid only for the microarchitectural latencies it was compiled against. If D_VPU, any FU latency, or N changes, the kernel must be recompiled. The compiler is the sole authority on timing.

7. Deferred / Notes

  • ACC bank-index operand: deferred (#313 / depth sweep #262); v1 uses bank 0 only.
  • Full 16-bit SPAD/ACC row addressing: the MXU drive path currently truncates to 13-bit (interim).
  • DMU shape packing: op2 packs both matrix dims as (rows-1)<<8 | (cols-1); dram_stride lives at pool[idx+1].
  • loop.begin lo/step: capped at 8 bits (0..255); spill hi-style if a kernel needs > 255.
  • N: N=8 is the baseline. N is a microarch parameter, NOT ISA-visible (16 is an ISA-invariant sweep target).
  • Implementation gate (not part of this spec): the branch's encoders.py + ISS still carry pre-0.5.0 semantics (acc.store direction, SPAD-X matmul, missing new-op encoders) and must be rewritten to match this spec before the ISA is merge-locked.
