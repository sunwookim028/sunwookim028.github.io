    Memory Budget: Qwen 2.5 0.5B GPTQ-Int4 on NPU¶

    Target model: Qwen 2.5 0.5B GPTQ-Int4
    Target hardware: Custom NPU (geometry SSOT mininpu/hwparams.py), 512 MB DRAM
    Audience: RTL architect and compiler engineer - exact DRAM / KV-cache sizing for the 512 MB budget. On-chip pool sizing is geometry-dependent and lives with the SSOT (see the geometry pointer at the end), not here.

    Model config¶

    Model config (confirmed from safetensors header):

    Parameter Value
    hidden_size 896
    num_layers 24
    num_attention_heads (Q) 14
    num_key_value_heads (GQA) 2 (group = 7)
    head_dim 64
    intermediate_size 4864
    vocab_size 151,936
    group_size (GPTQ) 128
    zero_point 7 (uniform)
    torch_dtype float16

    All F16 non-integer tensors confirmed from safetensors (no bf16).
    Bias on Q [896], K [128], V [128] projections; no bias on O, gate, up, down.

    DRAM budget (512 MB)¶

    This budget is a function of the model weights and KV-cache geometry only; it is independent of the on-chip scratchpad pool.

    Fixed model weight costs¶

    All weights are in the reformatted NPU layout: [N, K//2] uint8 for quantized weights, [N_groups, N] F16 for scales, [N] F16 for biases.

    Per-layer weight breakdown:

    Projection Shape (K, N) int4 bytes [N, K//2] Scales [groups, N] F16 Bias [N] F16 Total
    q_proj K=896, N=896 896×448 = 401 KB 7×896×2 = 12 KB 896×2 = 2 KB 415 KB
    k_proj K=896, N=128 128×448 = 56 KB 7×128×2 = 2 KB 128×2 = 0.3 KB 58 KB
    v_proj K=896, N=128 56 KB 2 KB 0.3 KB 58 KB
    o_proj K=896, N=896 401 KB 12 KB - 413 KB
    gate_proj K=896, N=4864 4864×448 = 2129 KB 7×4864×2 = 66 KB - 2195 KB
    up_proj K=896, N=4864 2129 KB 66 KB - 2195 KB
    down_proj K=4864, N=896 896×2432 = 2129 KB 38×896×2 = 66 KB - 2195 KB
    input_layernorm [896] F16 - - 2 KB 2 KB
    post_attn_layernorm [896] F16 - - 2 KB 2 KB
    Per-layer total 7,533 KB = 7.35 MB

    Full model:

    Component Size Notes
    24 × layer weights 176.1 MB All projections + norms
    embed_tokens / lm_head (tied) 259.7 MB [151936, 896] F16; single storage
    model.norm.weight 1.8 KB Final layernorm
    Fixed total 435.8 MB

    Available for KV cache¶

    512 MB total
- 435.8 MB fixed weights
= 76.2 MB available
- ~1.0 MB reserved for DRAM activation buffers and runtime scratch
  (hidden state x: 32×896×2 ≈ 56 KB; input/residual copy: ≈56 KB;
   FFN intermediate if spilled: 32×4864×2 ≈ 304 KB;
   runtime overhead stack/descriptors: ~256 KB → total ≈700 KB, rounded up to 1 MB)
= 75.2 MB for KV cache

    KV cache per token¶

    Each token appends one K vector and one V vector per layer, per KV head:

    KV per token = 24 layers × 2 (K+V) × 2 KV-heads × 64 head_dim × 2 bytes (fp16)
             = 24 × 2 × 2 × 64 × 2
             = 12,288 bytes = 12 KB per token

    Maximum context budget (512 MB DRAM, fp16 KV)¶

    Max context tokens = floor(75.2 MB / 12 KB) = floor(75.2×1024×1024 / 12288) = 6,417 tokens

    Context / output generation budget:

    Input tokens Max generated tokens Total context
    512 5,905 6,417
    1,024 5,393 6,417
    2,048 4,369 6,417
    4,096 2,321 6,417
    6,000 417 6,417

    Note: Qwen 2.5's max_position_embeddings = 32,768, but 512 MB DRAM limits the hardware context to 6,417 tokens at fp16 KV precision (after reserving 1 MB for DRAM activation buffers).

    Context extension options¶

    KV dtype Bytes/token Max context Notes
    fp16 (default) 12,288 6,417 No quality loss
    int8 quantized 6,144 12,834 ~0.5% perplexity increase
    int4 quantized 3,072 25,668 ~1-2% perplexity increase

    All figures use 75.2 MB KV budget (76.2 MB − 1 MB activation overhead). Formulae: floor(75.2×1024×1024 / bytes_per_token).

    For int8/int4 KV, additional dequant ops are needed during attention (KV fetch → dequant → fp16 for matmul). The NPU ISA already has dma.load.spm.i8 and vec.dequant.i4 which can be extended for this purpose.

    Hardware geometry¶

    Hardware geometry: see the SSOT mininpu/hwparams.py (DIM=8, one 32 KiB unified pool).

    Model / precision facts (recovered from the removed ARCH_SPEC_NPU.md)¶

    docs/ARCH_SPEC_NPU.md was deleted and this file was trimmed; the following precision-contract facts for the Qwen 2.5 0.5B target survived only in working memory and are restored here. The model config already documented above (hidden=896, 24 layers, 14 Q : 2 KV group-7 heads, head_dim=64, inter=4864, vocab=151936, group_size=128, zero_point=7, float16, bias-on-QKV) is unchanged.

    Positional / norm constants (host-side): - rope_theta = 1_000_000 (NOT 10_000) - host-side cos/sin table parameter. - rms_norm_eps = 1e-6. - tie_word_embeddings = True - embedding and lm_head share one weight, stored once.

    GPTQ host-side offline repack (done before DMA, not on-NPU): - Weights: on-disk [K//8, N] int32 (8 nibbles per int32) -> repacked to [N, K//2] uint8 for NPU DMA. - Zeros: qzeros [N_groups, N//8] int32 -> unpacked to [N_groups, N] fp16 (all values = 7).

    Accumulation precision: - fp32 accumulation is MANDATORY for the large contraction dims (K=896, K=4864). fp16 accumulation error at K=4864 is ~5x the fp32 path.

    Op-lowering notes: - Bias-add is implemented as dma.load.spm(dram_stride=0) broadcast + cast.f16.f32 + vec.add.acc. - lm_head reuses matmul.tile - no new opcode is introduced for it.

    For how these lower through the backend see COMPILER_ARCHITECTURE.md; numeric tolerance contracts are the sibling ISA's § Numerical Precision Contracts in ../isa/npu-isa.md.
