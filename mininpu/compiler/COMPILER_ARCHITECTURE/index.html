    Compiler Architecture¶

    End-to-end walkthrough of the npu-compiler backend: how a Triton kernel becomes NPU assembly, and the structural constraints that bound what is lowerable. This is the architecture SSOT; it cross-links (does not duplicate) the surface contract in SUPPORTED_TRITON.md, the milestone ladder in charter.md, and the ISA narrative in ../isa/npu-isa.md (opcode/encoding SSOT: sibling mininpu/isa/opcodes.py).

    1. Pipeline¶

    Triton (.py)
  -> triton-opt + triton-shared-opt          (triton-shared MLIR frontend)
2_linalg.mlir
  -> npu-opt --linalg-to-npu                 (LinalgToNPU.cpp)   [only conversion pass]
  -> npu-opt  PhysAllocPass                   (physical bank / pool assignment)
3_npu.mlir
  -> npu-translate --to-asm                   (NPUToAsm.cpp)      [serialization]
4_npu.s
  -> mininpu.asm parse_asm() -> mininpu.iss.simulator Simulator

    mininpu_compiler/compile.py :: compile_kernel(fn, tiles) drives all stages. Per-stage snapshots are checked in under test/e2e/pipeline/{matmul,flash_attn}/. These .s snapshots are geometry-dependent and compiled-in (see §5): regenerating them requires a C++ rebuild, not just re-running the Python driver.

    2. Two-level tiling model¶

    Tiling is expressed at two levels and they must agree:

    • Program tiling - the Triton grid / tiles argument chooses the per-program tile shape and the number of program IDs (pid). Loop trip counts derived from problem size are literal in the emitted .s and are patched host-side per-pid (see §6).
    • Hardware tiling - each program's tile is further decomposed to the DIM x DIM systolic array and the 32 KiB unified pool. This decomposition is fixed by the compiled-in geometry (§5), not by the kernel.

    Arbitrary problem sizes are handled by looping the fixed hardware tile, not by re-parameterizing the array.

    3. Conversion: LinalgToNPU (the only conversion pass)¶

    LinalgToNPU.cpp is the sole linalg->npu conversion pass. Its patterns are deliberately narrow:

    • Almost every pattern calls getSingleBodyOp() and matches exactly one arith/math op in the region body. Single-op elementwise/reduction bodies lower cleanly through the pattern set.
    • Multi-op bodies bypass the clean patterns. Fused activations whose body is several ops (GELU, LayerNorm, SiLU/SwiGLU, RMSNorm) do not match a single-op pattern and are instead reconstructed later in the serializer's fold matcher (§5). This is the root of the extensibility debt tracked in issues #10-#12: GELU is the same class as SiLU; LayerNorm is RMSNorm's fold extended with a mean-subtraction term.

    4. Address calculation: LinearAddrAnalysis¶

    Address generation lives in LinearAddrAnalysis.cpp (renamed from the older LinearAddr analysis). It computes linearized DMA/pool addresses from loop induction variables.

    Hard constraint: induction-variable nesting is capped at iv <= 1. Address generation supports at most one nested induction variable, which bounds the loop-nest depth an address expression can span. New ops or tilings that would require deeper affine address nesting are not lowerable without extending this analysis. Treat iv <= 1 as a first-class limit when scoping new kernels.

    5. Physical allocation + serialization¶

    PhysAllocPass assigns physical banks / pool offsets. Geometry is compiled into the binary: PhysAllocPass::getPhysProfile() hardcodes named profiles (historically tutorial dim4 / bench_n8 / prod dim16). Changing geometry therefore requires editing C++ and make build, and then regenerating the pipeline .s snapshots. The single fixed shipped geometry (DIM=8, one 32 KiB pool) is described in the root CLAUDE.md hardware-geometry table; mininpu/hwparams.py in the sibling package is the numeric SSOT.

    NPUToAsm.cpp serialization is a plain in-order IR-walk - there is NO instruction scheduler. Instructions are emitted in program order; there is no reordering, no software pipelining, and no double-buffering. The flags[5]=async DMA bit is not exploited (it is a functional no-op in the current sim). This is the central efficiency limitation; the design rationale and the planned scheduling + async-DMA pass are in charter.md § Efficiency (design notes).

    Two concrete debts inside NPUToAsm.cpp (context for issues #10-#12):

    • Stateful fold matcher (~360 lines, approx L709-1073). Multi-op activation bodies (§3) are reconstructed here via five order-dependent state maps: fillConsts, siluSrcOf, scaleAddOf, sqrtMarked, broadcastSrc. This rebuilds SiLU/RMSNorm from the serialized stream. Adding GELU/LayerNorm means extending these state maps, not adding a clean pattern.
    • Mirrored dyn_cast chains. translateOp and walkOp each carry ~23 hand-mirrored dyn_cast branches that must be kept in sync manually (the file header admits this). #12 proposes generating them from NPU.td.

    6. Host-patch seam (loop bounds)¶

    Loop trip counts are literal-only in the emitted assembly. Problem-size- and pid-dependent bounds are resolved by the host at launch via an ABI patch table: the host writes per-pid loop bounds / kernel-arg CSRs before dispatch (e.g. loop.begin.csr reads kernel_args[csr_idx]). The compiler does not emit dynamic bound arithmetic; it emits placeholders + a patch descriptor. This is why a wrong or omitted compile-time scalar (e.g. the VALID_CTX / N_CTX_TILES class of bug, issue #6) silently lowers to an unset runtime arg rather than a compile error.

    7. Structural limits (what bounds new work)¶

    • Single-op body matching in LinalgToNPU (§3) - multi-op ops fall to the fold matcher.
    • iv <= 1 address-nesting cap in LinearAddrAnalysis (§4).
    • Geometry compiled-in via PhysAllocPass (§5) - geometry changes need a rebuild + snapshot regen.
    • No scheduler / no double-buffering / async-DMA unused (§5) - correctness-first, efficiency deferred to v2.

    8. Packaging & external consumption¶

    Outside this repo the compiler is consumed, never built from source: consumers install a self-contained wheel (scripts/build-release.sh), which decouples them from the heavy MLIR build and the Triton pin. Release facts worth preserving:

    • Default deliverable is a linux_x86_64 wheel (~40 MB stripped, fits PyPI's 100 MB limit): npu-opt/npu-translate statically link MLIR/LLVM/Triton, bundle libstdc++/libgcc_s, and set RPATH=$ORIGIN/../lib (zero LD_LIBRARY_PATH).
    • No manylinux tag is emittable. An earlier setup.py hand-asserted manylinux_2_28; that was FALSE - auditwheel proves the binaries require GLIBC_2.34 and are x86_64_v2, so a 2_28-tagged wheel installs on RHEL8 and then crashes. auditwheel is the truth-teller, not a gate. A manylinux build is only reachable via local docker (scripts/build-release-docker.sh) + auditwheel repair. The [frontend] extra pins triton==3.4.0.
    • Ecosystem (github.com/sunwookim028), for cross-repo context only: mininpu = RTL/verification/FPGA and the Python package that is the ISA SSOT (file ISA issues there; Discussions are OFF, use Issues); mininpu-sdk/mininpu-runtime = host driver (NpuDriver); mininpu-compiler = this repo; mininpu-inference (ex mininpu-software) = Triton kernel suite + LLM integration + tutorial arc; mininpu-labs is RETIRING (absorbed into mininpu). Triton is pinned to 3.4.0 ecosystem-wide (triton-shared built against triton e44bd1c8).

    9. Design positioning (production analogs)¶

    The backend's structural choices are the production-default for accelerator compilers, not minimal shortcuts. The named analogs are the rationale for each shape:

    • CISC matmul.tile macro over RISC-shaped vector ops mirrors the TPU (one MXU matmul opcode wraps the inner DIM-loop; surrounding ops are RISC) and Triton's tl.dot. The tile is the hardware loop unit, so the inner array loop is never exposed to software. flash_attn.tile-style CISC macros are a logical extension of the same design.
    • Three on-chip spaces (SPM bf16 / ACC fp32 / VREG) match the TPU's VMEM/SMEM/CMEM split. Three is the production minimum: fewer would force precision conversions, more would need extra DMA. fp32 accumulation is required for numerical stability across long K. (These are dtype views of one unified byte-pool, per §5.)
    • Out-of-tree MLIR npu dialect (§3, §5) is the production-default for new accelerator compilers - IREE, Triton, Mojo, Cerebras CSL, and Tenstorrent TT-Forge are all out-of-tree MLIR - reusing the upstream pattern rewriter, type system, and linalg/scf/memref dialects.
    • Online softmax decomposed at the linalg level (explicit rowmax / sub / exp2 / scale / fma generics) matches how Triton, JAX Pallas, and Mojo express attention; the decomposed form is the right granularity for a vector ISA. The absence of RISC-level fusion across those vector ops is the efficiency gap in charter.md § Efficiency (issue #25); the CISC-macro answer is the other direction on the same axis.
