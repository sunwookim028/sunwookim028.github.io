    mininpu-compiler - Charter¶

    The single source of truth for this project's milestone ladder and its key design decisions. Roadmap-style planning lives here; current defects/gaps are tracked as GitHub issues; the compile surface lives in SUPPORTED_TRITON.md.

    Purpose¶

    A minimal, extensible compiler from Triton to a custom NPU ISA. The ISA is deliberately small - target band 20-50 instructions (still being defined; see the SSOT below) - and every workload must be expressible as tiled loops over DIM×DIM matmul + vector primitives. The compile target is transformer inference kernels: GPT-2-style blocks (LayerNorm, GELU, standard MHA) and the LLaMA family (RMSNorm, RoPE, SwiGLU, GQA/FlashAttention), composed into end-to-end decode.

    Milestone versions¶

    Two independent version axes - do not conflate them. This milestone version (v0/v1/v2/…) tracks compiler capability. It is distinct from the ISA version axis (the sibling package's ISA_VERSION, currently 0.4.0, and the spec-narrative "v1.x freeze" codename). A milestone vN may ride on any ISA_VERSION.

    v0 - DONE (baseline)¶

    Triton → NPU end-to-end for tiled GEMM, FlashAttention forward, and Qwen 2.5 0.5B greedy decode. Folds the historical milestones M0-M5c (see appendix). The current tree is on the single-geometry re-sync (#9): one fixed DIM=8 geometry, no NPU_PROFILE dual profile. Full end-to-end re-verification is currently gated on the sibling ISA settling (ISA_VERSION is in flux and mininpu.asm does not import cleanly today), so v0 is "done by construction, pending a green re-run". The sibling mininpu package is under active development right now (ISA rework in progress); sibling-coupled actions - the mininpu dependency-pin finalization, end-to-end re-verification, and pipeline-snapshot regeneration - are deliberately deferred until the sibling ISA settles. The mininpu>=0.4.0 pin in pyproject.toml is therefore provisional.

    v1 - NOW: GPT-2 op coverage + compiler-architecture prework¶

    • Op coverage: LayerNorm, GELU, residual-add, standard (non-GQA) MHA. The SSOT already reserves vec.gelu; the dialect does not yet emit it (tracked in #11).
    • Compiler prework this requires:
    • A real multi-op elementwise-body lowering strategy (retire the ad-hoc translator side-tables; land SiluAccLowering and friends in LinalgToNPU).
    • Restore the ISA-coverage drift test (currently xfail'd, see #11).
    • Consider generating the two mirrored dyn_cast chains (PhysAllocPass ↔ translateOps) from NPU.td instead of hand-syncing them.

    Details and acceptance criteria live in the v1-labeled GitHub tracking issues (umbrella #21).

    v2 and beyond - TBD¶

    Scope not yet defined. Candidate directions (unscheduled, not committed to a version): FPGA/ZCU102 deployment, FP8 execution path (Phase 2 ISS handler / Phase 3 driver), broader model-family kernels.

    Design decisions (SSOT)¶

    • ISA SSOT = sibling mininpu/mininpu/isa/opcodes.py (the frozen opcode / field / flag / capacity table + precision contracts). Do not hard-code the opcode count in docs - read opcodes.py (currently 38 opcodes, RISC-style; target band 20-50). isa/npu-isa.md is non-normative narrative. A CISC-style variant is under active consideration - the project may migrate to CISC or evaluate RISC and CISC side by side - which would reshape the compiler's lowering granularity (relevant to v1 prework); this is an open design direction, not a decision.
    • Hardware-geometry SSOT = sibling mininpu/mininpu/hwparams.py: a single fixed geometry, DIM=8, one 32 KiB unified on-chip pool (SPM/ACC are dtype views of the same byte-pool, not separate budgets). No NPU_PROFILE.
    • Tiling model: scf.for → loop.begin/loop.end + matmul.tile; loop bounds are literal-only and patched host-side (per-pid via the ABI patch table); all shapes are tile-aligned (host pads partial tiles).
    • Milestone-tracking policy: this charter defines the ladder; execution is tracked in GitHub issues labeled v0/v1/v2/…. This file replaces the retired roadmap doc.

    Efficiency (design notes) - why v2 exists¶

    The v0/v1 work above establishes that the target kernels (arbitrary-size tiled GEMM, FlashAttention forward, Qwen decode) are expressible. They are not yet efficient. v2's reason to exist is closing that gap; the FPGA/FP8/model-family candidates listed under v2 are downstream of it. See COMPILER_ARCHITECTURE.md §5 for where these live in the backend.

    Four structural gaps:

    1. No instruction scheduler / no double-buffering. Serialization (NPUToAsm.cpp) is a naive in-order IR-walk: no reordering, no software pipelining, DMA and compute strictly sequential.
    2. The 32 KiB unified pool is not reused across kernels - no cross-kernel residency planning, so operands are re-DMA'd.
    3. Non-matmul (vector/elementwise) ops are under-served by the pipeline relative to the matmul path.
    4. DIM=8 systolic array is small (hardware ceiling, not compiler work).

    Levers:

    • A scheduling + async-DMA pass that actually exploits flags[5]=async (today a functional no-op) to overlap DMA under compute and enable double-buffering. Tracked as the v2 efficiency issue.
    • The RISC-vs-CISC ISA-granularity choice reshapes lowering granularity and the achievable overlap, so it is coupled to the ISA revision axis.

    Cost / geometry design rationale (measured)¶

    These findings come from a cycle cost-model (sim/cost/) and cost snapshots that were later removed from git (commit 116b2b1); the measured conclusions that justify the shipped geometry are recorded here so they are not lost. Caveats: the cost model had ZERO ISS coupling (programs were hand-translated, not compiled .s), and flags[5]=async was a sim no-op, so absolute numbers are directional.

    • 2 DMA channels land within ~20% of 4 under per-burst arbitration -> the proposal went 4ch -> 2ch fair-share AXI.
    • Post-Gap1+2 the regime is compute-bound (async DMA fully hides under compute) - which is exactly why the missing scheduler/async path (above) leaves throughput on the table.
    • Target systolic width is fp16 DIM_FP=8 (~192 DSP on ZCU102) vs the current bitstream DIM=4. At M=1 decode, DIM=4 is ~1.75x the cycles of DIM=8 (not 4x).
    • vec_lane is the surprise bottleneck: 16 -> 32 buys ~+21% throughput.
    • GQA fusion saves ~5.9x DRAM bytes but ~0 wall-clock when compute-bound.
    • Tokens/sec trajectory: status-quo 0.078 -> Gap1+2+GQA+vl32(2ch) 0.136 (~1.74x); ASIC ceiling 0.883. Hotspots: decode_attn 53%, gate/up/down 41%.

    Net: the shipped choices (DIM=8, vec_lane=32 target, 2-channel DMA) are the cost-model sweet spot, and the biggest un-banked win is the scheduling/async pass.
