    Dependencies¶

    This project has three install tiers, from lightest to heaviest. Pick the tier that matches what you need to run; each one is a strict superset of the previous.

    Dependency SSOT: pyproject.toml ([project.dependencies] + [project.optional-dependencies]). requirements.txt is a thin convenience mirror for pip install -r requirements.txt; it is kept in sync by hand and must never diverge from pyproject.toml (same versions, same triton pin).

    Python requirement: >=3.9 (all tiers).

    Tier 0: the mininpu sibling package¶

    Everything below depends on the mininpu package: the functional simulator / assembler / ISA SSOT (mininpu.iss, mininpu.asm, mininpu.isa, mininpu.hwparams).

    mininpu is not published to PyPI. It lives in a sibling repo and is expected at ../mininpu relative to this repo's checkout:

    git clone https://github.com/sunwookim028/mininpu ../mininpu   # if not already cloned
pip install -e ../mininpu

    pip install -e . in this repo declares mininpu>=0.4.0 as a dependency, but pip cannot resolve it from PyPI. Install the sibling first, or install this package with --no-deps after mininpu is already present:

    pip install -e ../mininpu
pip install -e . --no-deps

    mininpu runtime subset (internal distribution)¶

    Distribution is internal (a small shared group, not public PyPI). The compiler ships as a self-contained wheel; the mininpu runtime it depends on ships alongside it as a trimmed subset, not the whole mininpu repo.

    The shipped mininpu_compiler package has exactly one hard mininpu import: compile.py does from mininpu.asm import assemble_program (the binary-image path). Everything else importing mininpu is tests/demo, not the shipped package.

    Verified runtime import closures:

    • import mininpu.asm -> {asm, core, isa}
    • import mininpu.iss.simulator -> {iss, core, isa, hwparams}
    • Neither loads mininpu.cost (its cost imports are lazy, inside functions).
    Module Status Reason
    mininpu/__init__.py REQUIRED package root
    mininpu/hwparams.py REQUIRED hardware geometry SSOT (sim path)
    mininpu/isa/ REQUIRED opcode/encoding SSOT
    mininpu/core/ REQUIRED insn/decode core
    mininpu/asm/ REQUIRED assembler (assemble_program)
    mininpu/iss/ REQUIRED functional simulator
    mininpu/cost/ EXCLUDED not imported by this repo (lazy-only)
    mininpu/abi.py EXCLUDED not needed by the compiler
    mininpu/ip_packaging.py EXCLUDED not needed by the compiler
    everything else (RTL, verification, docs) EXCLUDED not runtime

    Two tiers within the subset:

    • compile-only needs {isa, core, asm, hwparams}
    • + functional simulation adds {iss}
    • cost-model features are intentionally unavailable in the subset.

    Stage it with:

    bash scripts/stage-mininpu-subset.sh
#   MININPU_SRC  source checkout   (default: sibling ../mininpu)
#   DEST         staging dest      (default: <repo>/dist/mininpu-subset)

    The script copies only the required subset, runtime-only: it excludes __pycache__/*.pyc and mininpu's internal tests (tests/ dirs and test_*.py files), then runs a static import-closure check. Dropping the tests also removes the only (lazy) mininpu.cost reference, so the artifact has no cost coupling at all. The shared-server destination is TBD; for now it stages into a local dist/ the release process copies onward.

    Tier 1: sim-only (no compiler build)¶

    Runs the Python functional simulator and unit tests. No MLIR, no C++ toolchain.

    pip install -e ../mininpu
pip install numpy pytest      # or: pip install -e .[dev] --no-deps
make sim-test

    Tier 2: compiler path (npu-opt, npu-translate, e2e tests)¶

    Requires a prebuilt MLIR tree at the pinned LLVM commit.

    bash scripts/build-mlir.sh    # clones llvm-project @ pinned commit, ~20 min
make build                    # cmake + ninja: npu-opt, npu-translate, ~1 min
make e2e-test                 # test/e2e/sim_validate (needs npu-opt)

    triton-shared-opt (needed for the Triton frontend, tier 3) is built separately:

    make build-triton-shared      # ~5 min, after build-mlir

    Tier 3: Triton frontend (.py kernel -> linalg -> NPU asm)¶

    Only needed for make triton-test (compiling actual Triton kernels, not just running pre-lowered snapshots). Pulls the pinned Triton frontend.

    pip install -e .[frontend]    # triton==3.4.0
make triton-test

    Triton pin: the only Triton version anywhere in this repo is ==3.4.0 (the [frontend] and [dev] extras in pyproject.toml, mirrored in requirements.txt). triton-shared-opt is built against Triton commit e44bd1c8 == 3.4.0; do not bump one without the other.

    Optional: demo extra¶

    demo/chat.py and scripts/demo_tinyllama.py load HuggingFace weights and tokenizers. torch is intentionally excluded from this extra: weights load via safetensors directly as numpy; torch is only used for the optional HF reference-output comparison.

    pip install -e .[demo]        # transformers, safetensors, tokenizers

    The full demo conda-env setup (bash scripts/setup-demo-env.sh) additionally installs CPU-only torch and pre-downloads model checkpoints; see that script for the full package list.

    Quick reference¶

    Goal Command
    Sim unit tests pip install -e ../mininpu && pip install -e .[dev] --no-deps && make sim-test
    Compiler e2e tests above + bash scripts/build-mlir.sh && make build && make e2e-test
    Triton atom-kernel tests above + make build-triton-shared && pip install -e .[frontend] && make triton-test
    Demo (chat) pip install -e .[demo] (or bash scripts/setup-demo-env.sh for the full conda env)
