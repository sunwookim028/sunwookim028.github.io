    Quality Audit: mininpu-compiler¶

    Scope: the compiler-side repo only (Triton to custom NPU ISA, MLIR based). The sibling mininpu ISS/simulator package and its hwparams.py/opcodes.py SSOTs are out of scope and are referenced only as the authority the compiler must agree with.

    Executive summary¶

    • The compiler tree has a genuinely clean, recognizable out-of-tree MLIR skeleton: TableGen npu dialect, include/lib/tools layering, npu-opt as an MlirOptMain driver, npu-translate as an MlirTranslateMain hook, idiomatic add_mlir_library targets, and an excellent "why" comment culture (W9/D2b/D4 rationale).
    • Two issues rise above the rest and should be fixed first:
    • CI is broken: the e2e-tests job calls a nonexistent make triton-test-tutorial target (blocker), and both jobs pip install a mininpu dependency that is not resolvable on a stock runner (high).
    • A latent correctness hazard: PhysAllocPass::getPhysProfile() still implements the removed NPU_PROFILE mechanism and defaults to DIM=16 / VREG_COUNT=32, contradicting the DIM=8 / VREG_COUNT=8 hardware SSOT, so the allocator can hand out physically-invalid VREG indices.
    • The strongest "not yet a real MLIR project" signals are structural, not buggy: no lit infrastructure (a hand-rolled bash FileCheck runner stands in and is not run in CI), and real compiler work (physical allocation, linear-address analysis) living inside the translation library instead of as npu-opt passes, producing a ~1494-line translator with self-admitted parallel dispatch chains.
    • The dialect is small and readable, but its defining invariant (three memory banks: ACC = rank-2 f32, SPM = rank-2 f16, VREG = rank-1 f32) is enforced by hand-written C++ verifiers on only 5 of 26 ops while every operand is the loose AnyMemRef. This is the top MLIR-idiom improvement available.
    • Documentation is well organized with real SSOT discipline, but has a verifiable staleness cluster around decode-step attention (shipped and tested, still documented as unimplemented/planned in two files), a fully fictional README demo/chat.py section, and several stale GLOSSARY.md cross-references and counts.
    • The bulk of the remaining backlog is low-risk hygiene, doc accuracy, and safe code cleanups that can be auto-applied.

    Deployment posture (clarified 2026-07-08)¶

    The ci.yml:29 finding above and the "published wheels" option in the deferred appendix were written assuming a public-PyPI target. That is not the actual deployment model, and the finding's severity/framing should be read accordingly:

    • Distribution is internal only: a minimal group of about 50 users, via an internal/shared-server artifact (a self-contained linux_x86_64 wheel or the bundled binaries), not public PyPI.
    • The sibling mininpu runtime is likewise not published to PyPI; only the compiler-needed subset is shared internally alongside this package.
    • Because of this, manylinux/PyPI-tag honesty work is not a project goal, and the mininpu resolution gap in the public GitHub Actions CI job (which cannot reach an internal source on a bare runner) is a deprioritized optional mirror, not a release blocker. The internal artifact is smoke-tested against mininpu installed from source (see scripts/smoke-wheel.sh), which is the path that matters for real distribution.
    • Concrete mininpu subset decided: the compiler ships only the runtime it needs (isa, core, asm, iss, hwparams), excluding cost/, abi.py, and ip_packaging.py. scripts/stage-mininpu-subset.sh operationalizes this (verified import closure; local dist/ staging today, shared-server path TBD).

    These require judgement, cross-repo coordination, or architectural change and should NOT be auto-applied:

    • CI mininpu dependency resolution (ci.yml:29): needs a decision on how the sibling repo is made installable in the public GitHub Actions CI job (checkout + editable install, or an internal git/index source). Not a public wheel: see "Deployment posture" above. Deprioritized: internal distribution does not depend on this CI job. Cross-repo.
    • Geometry SSOT reconciliation (PhysAllocPass.cpp:52, NPUToAsm.cpp:66-67, PhysAllocPass.cpp:154): removing NPU_PROFILE and sourcing DIM/SPM_ROWS/ACC_ROWS/ VREG_COUNT from hwparams.py (ideally a generated header) changes emitted allocation behavior; verify against the frozen ISA/sim before landing.
    • translateOp decomposition (NPUToAsm.cpp:255) and hoisting allocation/addr-analysis into real npu-opt passes (NPUToAsm.cpp:28, PhysAllocPass.cpp:220): large refactors that alter the pass/translation boundary; land incrementally with FileCheck coverage.
    • Dialect type-constraint rework (NPU.td:36), per-operand memory effects (NPU.td:28), and the causal-attribute SSOT decision (NPU.td:41): behavioral ODS changes; sequence after they unblock the hasVerifier coverage work.
    • lit adoption (run-filecheck-tests.sh:1) + negative FileCheck coverage (linalgtonpu.mlir:1) + Passes.td/generated docs (CMakeLists.txt:1): the "real MLIR project" structural upgrade; best done as one coordinated change.
    • linalg.generic matcher relocation into LinalgToNPU (NPUToAsm.cpp:46): resolves the layering leak but changes where lowering failures surface; needs owner sign-off.
    • Driver unit-test suite (compile.py:46) and the profile param rename/removal (compile.py:321): low risk individually but touch public-ish CLI/test surface.
