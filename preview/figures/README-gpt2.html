    GPT-2 Kernel Schedules: The Numbers SSOT, and the Retiring HTML Renderer¶

    _gen_gpt2.py in this directory is the single source of truth for the GPT-2 per-kernel cycle numbers (timing-rev pin: ISA 0.5.0, source origin/rtl/full-npu-wire). It has two outputs, and they are at different points in their lives.

    1. tikz/gpt2-cycles.tex --- the generated LaTeX macro include (current)¶

    The kernel schedules are being ported to TikZ so they share the house figure template (tikz/fig-timing.sty) with every other figure on the site, and are embedded inline in the sections of hardware/microarch/compute_tile/gpt2_demo.md that already discuss each kernel.

    Porting to hand-drawn TikZ would normally cost the single-source property the HTML generator had. It does not, because the numbers stay generated: this script emits tikz/gpt2-cycles.tex, a file of \newcommand definitions, and each figure \inputs it and draws macros rather than digits. No figure hand-types a cycle count. A number changes here and every figure that uses it redraws.

    make gen-gpt2-cycles     # regenerate the include (also rewrites the legacy HTML)
make check-gpt2-cycles   # drift gate: fails if the committed include is stale

    The drift gate is the same mechanism as the generated ISA/ABI reference fragments (docs/scripts/gen_isa_abi_md.py --check): a --check mode plus a pre-commit hook (gpt2-cycles-check). CI's pre-commit job runs pre-commit run --all-files, so the hook is not path-filtered and cannot be dodged by keeping the generator out of a diff. The generator additionally asserts each kernel's derived milestone sum against its declared span, so a constant edited without its total is an error, not a silent skew.

    gpt2-cycles.tex is source, not a page: mkdocs.yml's exclude_docs already carries figures/tikz/*.tex, so it is never published.

    Macro naming: \gptCy<Const> for shared timing constants, \gpt<Kernel><Milestone> for a per-kernel cycle. LaTeX control sequences are letters only, so the kernel key is CamelCased with the leading gpt2_ dropped (gpt2_qkv_proj -> QkvProj).

    2. gpt2_*.html --- the standalone Gantt pages (RETIRED)¶

    All seven are gone. They were the original renderer's output (self-contained inline-SVG standalone HTML), and they were orphans: nothing linked them, and they published to the site as pages reachable only by direct URL. Each is now superseded by a TikZ figure embedded inline in gpt2_demo.md:

    was is now
    gpt2_qkv_proj.html tikz/gpt2-qkv-proj.svg
    gpt2_qk_scores.html tikz/gpt2-qk-scores.svg
    gpt2_softmax.html tikz/gpt2-softmax.svg
    gpt2_layernorm.html tikz/gpt2-layernorm.svg
    gpt2_cproj.html tikz/gpt2-cproj.svg
    gpt2_residual.html tikz/gpt2-residual.svg
    gpt2_lmhead.html tikz/gpt2-lmhead.svg

    Their encoding was also no longer house style: six domain hues plus a dark-mode inversion is exactly what the figure template retired in favour of near-black ink on white with grey for memory. The TikZ ports do not carry it over.

    The renderer itself is kept and still works, gated by a PORTED set in _gen_gpt2.py. That set is what makes the deletion stick: without it the next generator run would recreate all seven files. Every kernel currently ported is listed there, so the renderer writes nothing today --- it is retained so a future kernel can be prototyped as HTML before it is drawn, and so gelu has a renderer waiting when #385 unblocks it. Removing a kernel from PORTED re-enables its HTML; that is the only way to get one back.

    What is HELD, and why¶

    _gen_gpt2.py carries a spec for every GPT-2 kernel, but one is deliberately not written to disk, not committed, and emits no cycle macros:

    • gpt2_gelu models gelu as a single fused latency-serial op. Owner ruling #385 section 5 decomposes gelu under ISA 0.5.0, so that model, and every roll-up that depends on it, is known-wrong. The spec is retained in the generator only as the section skeleton for whoever recomputes it. Remove it from the generator's HELD set only after the spec is recomputed under the decomposition.

    No other kernel sums a gelu cost, so the rest are unaffected: gpt2_cproj names gelu as the separate next kernel but does not add its cycles. The block-level roll-up figure is blocked on the same ruling and is not drawn.

    Kernel order (the section order in gpt2_demo.md)¶

    1. gpt2_qkv_proj - attn QKV projection (MXU GEMM + bias) --- drawn, tikz/gpt2-qkv-proj.tex
    2. gpt2_qk_scores - attn Q @ K^T scores (scaled, causal) --- drawn, tikz/gpt2-qk-scores.tex
    3. gpt2_softmax - attn row-softmax chain (VREG-only) --- drawn, tikz/gpt2-softmax.tex
    4. gpt2_layernorm - LayerNorm stats + affine --- drawn, tikz/gpt2-layernorm.tex
    5. gpt2_gelu - MLP gelu activation (HELD, see above --- not drawn)
    6. gpt2_cproj - MLP down-projection GEMM --- drawn, tikz/gpt2-cproj.tex
    7. gpt2_residual - residual add --- drawn, tikz/gpt2-residual.tex
    8. gpt2_lmhead - vocab projection + flush/halt tail --- drawn, tikz/gpt2-lmhead.tex

    This is 8 specs, 7 of them sound. It is not 12: there is no spec for embedding, attention-V, MLP fc, or argmax, and none should be invented to reach a count.
